# U4A Workspace — 전체 분석 개요 및 HTML5 컨버전 전략

> 본 `.analy` 폴더는 **U4A Workspace (v3.6.3)** 의 전체 플로우·기능을 영역별로 분석한 설계 문서 모음이다.
> 최종 목적은 이 분석을 토대로 **AI가 새 폴더에 화면(UI) 레이어를 HTML5 기반으로 재구현(컨버전)** 하는 것이다.

---

## 0. 컨버전 범위 정의 (가장 중요)

| 구분 | 내용 |
|------|------|
| **유지** | 앱은 **계속 Electron 데스크톱 앱**으로 동작한다. Electron API, `@electron/remote`, `ipcMain`/`ipcRenderer`(IPC), Node 모듈(`fs`, `net`/Named Pipe, `child_process`, `ws`, `regedit` 등), Web/Node Worker, SAP 백엔드 통신(XHR), `BrowserWindow` 창·팝업 관리, Windows Registry 접근, 자동 업데이트, NSIS/PowerShell/VBS 스크립트 — **전부 그대로 재사용**한다. |
| **교체** | 현재 화면이 **SAP UI5** 로 그려진 부분 → **순수 HTML5 + CSS + 바닐라 JS** 로 재구현한다. 대상: UI5 컨트롤(`sap.m`, `sap.tnt`, `sap.ui.table`, `sap.ui.layout`, `sap.f`, `sap.ui.unified` 등), UI5 View/Fragment, `JSONModel`·데이터 바인딩, UI5 i18n, UI5 테마, `sap.m.Dialog`. |
| **제거(교체 후)** | UI 교체 완료 후 `lib/ui5` SDK 로딩(`sap-ui-bootstrap`) 및 `ws_settings.json`의 `UI5` 섹션 제거 가능. |

> ⚠️ "fetch/WebSocket/localStorage/브라우저 모달로 대체" 같은 제안은 **이 프로젝트의 범위가 아니다.** Node/Electron 백엔드 자원은 손대지 않는다. 오직 "UI5로 그려진 화면"만 바꾼다.

## 1. 프로젝트 개요

- **이름**: U4A Workspace (`com.u4a_ws3.app`), v3.6.3
- **목적**: SAP 시스템에 로그인하여 웹 애플리케이션(U4A 앱)을 개발하는 **통합 개발 IDE**
- **플랫폼**: Electron 14 (Windows x64, NSIS 설치)
- **UI 프레임워크**: SAP UI5 (`node_modules/U4A/WS/lib/ui5/v11071/`)
- **코드 에디터**: Monaco Editor (UI5와 분리되어 iframe으로 통합), 일부 ACE Editor
- **백엔드 통신**: SAP 서버와 XHR(`/wsloginchk`, `/dummycall` 등), Named Pipe(서버리스트·AI 연동)
- **규모**: 약 8,140개 파일 (JS 2,177 / HTML 163 / CSS 67 + 이미지/svg 다수)

## 2. 부팅 플로우 요약

```
electron/main.js (메인 프로세스)
  └─ BrowserWindow 생성 → www/intro3.html 로드
       └─ intro.js : 테마·초기설치·레지스트리 점검, 서버리스트 호출
            └─ www/ServerList_v2/ServerList.html : 서버 선택
                 └─ BrowserWindow(자식) → Login.html : SAP 인증
                      └─ ws30/ws10_20/index.html (메인 프레임)
                           ├─ IPCRENDERER.on('if-meta-info') 로 메타/서버/유저정보 수신
                           ├─ <script id="sap-ui-bootstrap"> 동적 삽입 (UI5 로더)
                           │    · data-sap-ui-theme / -language / -libs 주입
                           │    · Node require/module 충돌 회피(window.__node 백업/복구)
                           └─ attachInit → _data_sap_ui_oninit → _initRendering
                                └─ views/vw_main/view.js (ESM import)
                                     └─ sap.m.App.placeAt("content")  ← WS10/20/30 NavContainer
```

상세 시퀀스·분기조건·IPC 채널 목록은 **[01_부팅_아키텍처.md](01_부팅_아키텍처.md)** 참조.

## 3. 전역 아키텍처 핵심 (컨버전 시 반드시 보존할 계약)

- **`oAPP` 네임스페이스**: 메인 프레임의 모든 로직 루트. `oAPP.fn`(기능) / `oAPP.events`(이벤트) / `oAPP.common`(공통) / `oAPP.attr`(상태) / `oAPP.views`(뷰) / `oAPP.usp` / `oAPP.DATA` 로 구성.
- **`parent.*` 체인**: 자식 iframe/팝업이 부모 창의 전역(`parent.REMOTE`, `parent.IPCMAIN`, `parent.PATH`, `parent.FS`, `parent.WSUTIL`, `parent.oAPP.fn.*` 등)을 직접 참조. **이 `parent.oAPP.*` API 시그니처는 컨버전 후에도 유지**해야 디자인 미리보기 iframe 등이 동작한다. (→ [05](05_디자인영역.md))
- **IPC 채널**: `if-meta-info`, `if-globalSetting-info`, `if-extopen-url`, `if-send-action-{BROWSKEY}` 등 20+ 채널. **전부 유지.** (→ [01](01_부팅_아키텍처.md))
- **메시지 클래스**: SQLite 기반 `WsMsgCls`(`electron/lib/msg/`)로 다국어 메시지 텍스트 조회. **유지.**

## 5. UI5 → HTML5 매핑 레퍼런스 (전 영역 공통)

각 문서의 분석에서 실제 사용이 확인된 UI5 컨트롤과 HTML5 대체안을 통합 정리한다. **개별 화면별 구체 대체안은 각 문서의 "UI 컨버전 고려사항(B)" 표를 우선**한다.

| UI5 컨트롤 / 개념 | HTML5 대체안 |
|---|---|
| `sap.m.App` / `sap.m.Page` | `<div>` 레이아웃 컨테이너 + CSS Flex/Grid |
| `sap.m.NavContainer` (WS10/20/30 전환) | 컨테이너 `<div>` + CSS `display` 토글 또는 라우팅 함수 |
| `sap.tnt.ToolPage` / `ToolHeader` (WS20) | Flex 레이아웃 + 헤더 `<header>` |
| `sap.m.Bar` / `OverflowToolbar` (헤더·푸터) | `<header>`/`<footer>` + Flex, 오버플로우는 CSS |
| `sap.ui.table.TreeTable` (소스트리·UI트리) | `<ul>/<li>` 재귀 트리 (또는 jstree/Fancytree). **교체 난이도 최상** |
| `sap.m.Table` / `sap.ui.table.Table` | `<table>` + 가상 스크롤(대용량 시) |
| `sap.m.Dialog` | HTML5 `<dialog>` 네이티브 요소 + 백드롭 |
| `sap.m.Input` / `SearchField` | `<input type="text">` / `<input type="search">` |
| `sap.m.ComboBox` + `Sorter`(그룹) | `<select>` + `<optgroup>` |
| `sap.m.Button` | `<button>` |
| `sap.f.Card` / `DynamicPage` | `<section>` + CSS 카드 스타일 |
| `sap.m.MessageToast` / `MessageBox` | 커스텀 토스트/모달 컴포넌트 |
| `sap.m.BusyDialog` / `setBusy` | 오버레이 `<div>` + 스피너 CSS |
| `sap.m.Menu` (context menu) | 커스텀 컨텍스트 메뉴 `<div>` (position: fixed) |
| `sap.ui.model.json.JSONModel` + 데이터 바인딩 | 전역 상태 객체 + `setState/getState/observeState` 패턴으로 DOM 수동 갱신 |
| UI5 i18n (`data-sap-ui-language`, i18n 모델) | JSON 리소스 + `data-i18n` 속성 치환 (기존 `MSG/` 리소스 재사용) |
| UI5 테마 (`data-sap-ui-theme`) | CSS 변수 기반 테마 + `data-theme` 토글. 기준 비주얼 `sap_horizon`, 테마 5종(white/dark/purple/red/green). 토큰 계약·전환 API는 **[12_테마_컨버전_전략.md](12_테마_컨버전_전략.md)** 단일 출처. (기존 테마 개인화 저장은 Node `fs` 유지) |
| `sap.ui.core.HTML` (Monaco iframe 래퍼) | `<iframe>` 직접 사용 (Monaco 자체는 유지) |
| 라이프사이클 `onAfterRendering` / `placeAt` | DOM `connectedCallback`/`MutationObserver` 또는 명시적 렌더 호출 |

## 6. 컨버전 시 특별 주의 사항 (Cross-cutting)

1. **디자인 미리보기 iframe의 UI5 런타임 (가장 어려운 결정)**
   디자인 영역의 미리보기(`design/preview/`)는 **사용자가 만드는 "대상 앱"** 을 실제 UI5 컨트롤로 동적 인스턴스화·렌더링한다. 대상 앱 자체가 UI5라면 이 영역은 **iframe 내 UI5 유지(현상 유지)** 가 현실적이다. IDE의 디자인 화면(트리/속성패널)만 HTML5로 교체한다. (→ [05](05_디자인영역.md))

2. **`parent.oAPP.fn.*` / `parent.oAPP.attr.*` API 계약 보존 + 전역 `sap` 위임 (실전 확정 해결책)**
   미리보기 iframe·컨텍스트 메뉴 등 다수 모듈이 부모 창의 함수/상태를 직접 호출하므로 시그니처를 유지한다. 또한 **복사된 IDE 코드 곳곳이 bare `sap`을 참조**(원본 IDE가 UI5라 전역 `sap`이 있었음) → UI5 제거 시 `ReferenceError: sap is not defined`. **해결: 메인프레임 전역 `sap`을 미리보기 iframe의 `sap`으로 지연 위임**(`window`+`global` 양쪽). 이 코드들이 다루는 UI5 컨트롤은 iframe에 살아있기 때문. → 코드/상세 [11 플레이북 §2-1](11_컨버전_플레이북_처음시작.md)

3. **Monaco는 이미 UI5와 분리됨**
   `sap.ui.codeeditor.CodeEditor`는 미사용. Monaco는 `sap.ui.core.HTML`에 iframe HTML을 삽입하는 구조라 래퍼만 `<iframe>`으로 바꾸면 된다. (→ [04](04_WS30_USP_코드에디터.md))

4. **파일·설정 저장은 전부 Node `fs` 유지**
   패턴/스니펫(`default_pattern.json`/`custom_pattern.json`), 테마 개인화, suggestion, p13n 등 모든 개인화 저장은 `userData` 경로에 Node `fs`로 직접 기록한다. **서버 API로 바꾸지 않는다.**

5. **팝업은 "여는 방식(BrowserWindow)"은 유지, "내용물(UI5)"만 교체**
   대부분 팝업이 `BrowserWindow` + `index.html → iframe → frame.html` 구조. BrowserWindow 생성·IPC 데이터 전달은 유지하고, 팝업 내부가 UI5면 그 부분만 HTML5로 재구현. (→ [06](06_팝업.md))

6. **전역 `jQuery` 공급 주체 변경 (1단계에서 확인된 실측 이슈)**
   원래는 **UI5 부트스트랩이 전역 `jQuery`(및 `$`)를 제공**했다. UI5를 제거하면 이 전역이 사라지고, 더구나 Electron `nodeIntegration` 환경에서는 `<script src="jquery.min.js">` 로 로드해도 jQuery UMD가 `window`가 아닌 `module.exports`에 등록되어 전역 `jQuery`가 비게 된다. → 각 HTML 진입 파일에서 **번들 jQuery(1.11.3) 로드 시 `module`을 잠깐 숨겨 브라우저 전역 경로로 등록**시키는 보정이 필요하다(예: `index.html`에 적용). jQuery·jQuery UI에 의존하는 기존 코드가 많으므로 화면 변환 시마다 이 보정을 적용할 것.

7. **floatingMenu(ABAP Editor)는 UI5 전면 사용**
   `frame.html`이 자체 `sap-ui-bootstrap`을 삽입하고 `sap.m.IconTabBar/NavContainer` 등으로 UI 구성. Electron 부분(`handle.js`)은 유지, UI 부분은 교체 대상. (→ [08](08_라이브러리_플로팅메뉴_도움말_리소스.md))

8. **실전 확정 방법론 (요약 — 상세는 [11 플레이북](11_컨버전_플레이북_처음시작.md))**
   - **override 방식**: 원본 UI5 JS는 무수정 복사·로드하고, 맨 마지막에 신규 `*_html5_*.js`가 같은 이름 함수를 재정의(재정의 파일은 항상 뒤에 로드).
   - **모델 shim**: UI5 `JSONModel` → `getProperty/setProperty/setData/refresh()` + **`oData` getter** 를 가진 상태객체.
   - **시작 즉시 "기반 보정 4종"**(전역 `sap` 위임, 전역 `jQuery` 복원, `showCriticalErrorDialog` APP종료 방지, 모델 oData·refresh) — 안 깔면 미변환 화면의 엣지/에러 경로에서 반드시 터진다.

## 8. AI 컨버전 실행 가이드 (기능 문서 + 테마 문서 결합)

> 이 절은 **메타 안내**다. 개별 화면의 구현 사양은 각 기능 문서가, 색·모양·테마 전환은 [12_테마_컨버전_전략.md](12_테마_컨버전_전략.md) 가 담당한다. AI 는 이 둘을 **함께 읽고** 한 화면의 "기능 + 테마"를 자기완결로 컨버전한다.  
> **AI 에게 실제로 지시하는 방법·재사용 템플릿은 [13_AI_작업지시_가이드.md](13_AI_작업지시_가이드.md) 를 사용한다.** (작업 단위 끊기, 0단계 토큰 harvest, 지시문 템플릿, 가드레일, 진행 순서)

### 8.1 실행 원칙

- **각 기능 문서에는 `## AI 컨버전 실행 사양` 절이 있다.** 이 절은 시안·참조 출처 / 산출물 파일 트리 / 작업 순서(입력→산출→수용 기준) / 불변 제약 / AI 실행 프롬프트의 표준 형식을 따른다.
- **AI 는 "대상 기능 문서의 실행 사양" + "12번 테마 문서"를 함께 적용**해야 화면이 완성된다. 기능 매핑(UI5→HTML5)만으로는 색·테마가 비고, 테마만으로는 기능이 빈다.
- **테마는 [12_테마_컨버전_전략.md](12_테마_컨버전_전략.md) 가 단일 출처**다. 기준 비주얼은 UI5 `sap_horizon`, 테마는 색만 바꾸는 5종(white/dark/purple/red/green)이다. 토큰 계약(의미 토큰 키)과 `U4ATheme.apply(name)` shim 이 여기에 확정되어 있다. 각 기능 문서의 CSS 는 이 토큰만 소비하며 색을 하드코딩하지 않는다(드리프트 금지). 모양(radius/elevation)은 horizon 공통 고정이라 테마 간 동일하다.
- **Preview iframe(WS20 디자인 미리보기) 내부는 UI5 테마를 유지**한다(셸 테마 시스템을 iframe 안으로 침투시키지 않음 — 12번 §1.1).
- **기능 동등성(Functional Parity)** — 기능 매핑·테마뿐 아니라 **기존 기능·동작·단축키·검증·서버분기·엣지케이스를 하나도 빠짐없이 동등하게** 재현한다. 외형이 아니라 **행동**이 기준이다. 각 기능 문서의 "기능 동등성 — 기능 인벤토리" 절을 전수 추출·매핑·검증한다(누락 0).
