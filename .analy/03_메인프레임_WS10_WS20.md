# U4A Workspace v3.6.3 — 메인프레임 + WS10 + WS20 코어 분석

> 기준 파일 경로: `www/ws30/ws10_20/`  
> 분석 날짜: 2026-06-09  
> 컨버전 방향: Electron/Node 자원은 **전부 그대로 유지**, SAP UI5 UI 레이어만 순수 HTML5+CSS+바닐라 JS로 교체

---

## 1. 개요

### 메인프레임 구조

U4A Workspace 메인프레임은 **단일 Electron BrowserWindow** 안에서 `www/ws30/ws10_20/index.html` 을 로드하는 구조다.  
이 HTML은 SAP UI5 bootstrap을 동적으로 삽입(`index.js`에서 `<script>` 태그를 `document.head.appendChild`로 주입)하고, UI5 초기화 완료 콜백(`data-sap-ui-oninit = "_data_sap_ui_oninit"`)에서 렌더링을 시작한다.

```
Electron Main Process
  └─ BrowserWindow (index.html)
       ├─ index.js           — IPC 리시버, UI5 bootstrap 동적 삽입, 초기 뷰 분기
       ├─ views/vw_main/     — 로그인 전 진입 뷰 (ESM, sap.m.App+Page+Bar)
       │   ├─ view.js        — UI 선언 (ESM export)
       │   └─ control.js     — 이벤트/초기화 로직 (ESM export)
       └─ js/library-preload.js  — oAPP 글로벌 선언 후 모든 ws_*.js 순차 eval 로드, ws_main.fnWsStart() 호출
```

### WS10 / WS20 / WS30 페이지 개념과 전환

| 페이지 ID | 역할 | 전환 트리거 |
|---|---|---|
| **WS10** | 애플리케이션 검색 (홈) | 초기 로드, WS20→10 Back |
| **WS20** | 애플리케이션 편집 (Display/Change) | WS10에서 APPID 조회 후 Display/Change |
| **WS30** | USP(User-Specific Page) 편집 | WS20 내에서 별도 요청 |

페이지 컨테이너는 `sap.m.NavContainer("WSAPP")` 하나이며, `oApp.to("WS10" | "WS20")` 방식으로 페이지를 전환한다.  
전환 시 `parent.setCurrPage(toId)` 로 Electron 메인 프로세스에 현재 페이지를 알린다.

- **WS10 → WS20**: `oAPP.fn.fnOnEnterDispChangeMode(APPID, ISEDIT)` → `fnMoveToWs20()` → `NavContainer.to("WS20")`
- **WS20 → WS10**: `oAPP.fn.fnMoveToWs10()` — Lock 해제, 세션 정리, `NavContainer.back()`

## 2. oAPP 네임스페이스 구조

`library-preload.js` 에서 `window.oAPP` 를 선언하고 각 파일이 하위 네임스페이스를 채우는 패턴이다.

```js
// library-preload.js
window.oAPP = window.oAPP || {};
oAPP.global  = {};   // 현재 미사용 (예비)
oAPP.main    = {};   // ws_main.js — 초기화·생명주기
oAPP.fn      = {};   // ws_fn_01~05.js — 비즈니스 로직
oAPP.events  = {};   // ws_events.js, ws_events_01.js — 이벤트 핸들러
oAPP.common  = {};   // ws_common.js — 공통 헬퍼 (모델 CRUD, 메시지, Busy/Lock 등)
oAPP.attr    = {};   // 런타임 상태 저장소 (플래그, 타이머, UI 참조 등)
oAPP.ui      = {};   // (설계 예비)
oAPP.usp     = {};   // USP 전용
oAPP.DATA    = {};   // 임시 데이터
oAPP.wmenu   = {};   // WS10/WS20 window menu 인스턴스 참조
oAPP.views   = {};   // VW_MAIN 뷰 컨트롤러 (index.js에서 추가)
```

| 네임스페이스 | 담당 파일 | 역할 |
|---|---|---|
| `oAPP.main` | `ws_main.js` | 앱 생명주기 (fnWsStart, fnBeforeunload, 모델 초기화, 윈도우 이벤트) |
| `oAPP.fn` | `ws_fn_01~05.js`, `fnSuggestion.js`, `fnHmws.js` 등 | 렌더링·비즈니스 로직·유틸리티 |
| `oAPP.events` | `ws_events.js`, `ws_events_01.js` | 버튼·입력 이벤트 핸들러 |
| `oAPP.common` | `ws_common.js` | 모델 set/get, 메시지, Busy/Lock, 단축키, Dialog, 헤더 버튼 |
| `oAPP.attr` | 런타임 할당 | 플래그(isShortcutLock, bIsNwActive 등), 타이머, appInfo 참조 |
| `oAPP.wmenu` | `ws_fn_01.js` 내부 | WS10/WS20의 `sap.m.Menu` 인스턴스 참조 |
| `oAPP.views` | `index.js` | 로그인 전 진입 뷰(ESM 모듈) 참조 |

## 3. 메인프레임 렌더링

### 초기화 순서

```
index.js (IPC: if-meta-info 수신)
  → UI5 bootstrap 동적 삽입 (sap.m, sap.tnt, sap.ui.table, sap.ui.layout, sap.f, sap.ui.codeeditor)
  → _data_sap_ui_oninit() 호출
    → views/vw_main/view.js : ESM import → sap.m.App + sap.m.Page + sap.m.Bar 렌더링
    → control.js.onViewReady() → getUserInfo() 분기
        ├─ 로그인 전: Login.html iframe 삽입
        └─ 로그인 후: library-preload.js script 태그 삽입
              → oAPP 네임스페이스 초기화
              → ws_common.js, ws_events.js ... ws_main.js 순차 eval 로드
              → oAPP.main.fnWsStart() 호출
                  → sap.ui.getCore().attachInit(async fn)
                      → 모델 초기화, fnOnInitRendering(), fnOnInitP13nSettings()
```

### 루트 UI 트리

```
sap.m.App (id: "WSAPP" / class: u4aWsApp sapUiSizeCompact)
  sap.m.Page (id: "WS10")  ← WS10 화면
  sap.m.Page (id: "WS20")  ← WS20 화면
  (WS30은 별도 fn 으로 동적 추가)
```

`sap.m.App` 은 `sap.m.Page` 래퍼(`oRootPage`)에 담겨 `sap.m.App("oRootApp")` 에 추가되고, `oRootApp.placeAt("content")` 로 DOM 에 마운트된다.

### 헤더 (oRootPage.customHeader)

`sap.m.Bar` — `u4aWsBrowserDraggable` 클래스

| 위치 | 컨트롤 | 역할 |
|---|---|---|
| contentLeft | `sap.m.Image` | WS 로고 (25px) |
| contentLeft | `sap.m.Title` (id: u4aWsHeaderTitle) | "U4A Workspace - Main" |
| contentRight | `sap.m.Button` (sap-icon://less) | 창 최소화 → `CURRWIN.minimize()` |
| contentRight | `sap.m.Button` (id: maxWinBtn) | 최대화/복원 → `CURRWIN.isMaximized()` 분기 |
| contentRight | `sap.m.Button` (id: mainWinClose, sap-icon://decline) | 창 닫기 → `CURRWIN.close()` |

### 푸터 메시지 (floatingFooter)

WS10, WS20 각 페이지에 독립적으로 `sap.m.OverflowToolbar` 푸터를 구성한다.

```js
new sap.m.OverflowToolbar({
    content: [
        new sap.ui.core.Icon({ color: "{/FMSG/WS10/ICONCOLOR}", src: "{/FMSG/WS10/ICON}" }),
        new sap.m.Text({ text: "{/FMSG/WS10/TXT}" }),
        new sap.m.ToolbarSpacer(),
        new sap.m.Button({ icon: "sap-icon://decline", type: Reject, press: fnHideFloatingFooterMsg })
    ]
})
```

모델 경로 `/FMSG/WS10` 또는 `/FMSG/WS20` 의 `ISSHOW` 값에 `bindProperty("showFooter", ...)` 를 걸어 표시 여부를 제어한다.

## 4. 헤더 윈도우 메뉴 시스템

### 구조 개요

각 페이지의 상단에는 `sap.m.OverflowToolbar` 안에 `sap.m.HBox` 로 묶인 여러 `sap.m.Button` 이 배치된다.  
버튼 클릭 시 대응하는 `sap.m.Menu` 가 펼쳐지는 방식이다.

```
OverflowToolbar (u4aWsWindowMenuToolbar)
  HBox
    sap.m.Button [Extras/StyleClass/Utilities/Edit/System/Help/Test]  ← 메뉴 카테고리 버튼
      → press: ev_pressWMenu10 / ev_pressWMenu20
        → Menu.openBy(oButton)
  ToolbarSpacer
  [공통 헤더 버튼] ← fnGetCommonHeaderButtons()
```

### WS30 윈도우 메뉴 카테고리

WS30은 WS20의 서브셋으로, WMENU20(Utilities), WMENU30(System), WMENU50(Help), Test10 으로 구성된다.

### 메뉴 데이터 바인딩 패턴

```js
// 1. 데이터 생성
let oMenuList = oAPP.fn.fnGetWindowMenuListWS10();  // {WMENU10:[...], WMENU20:[...], ...}
oMenuList.HEADER = oAPP.fn.fnGetWindowMenuWS10();   // [{key:"WMENU10", text:"Extras"}, ...]
APPCOMMON.fnSetModelProperty("/WMENU/WS10", oMenuList); // Core JSONModel에 저장

// 2. 메뉴 객체 생성 (항목별 sap.m.Menu)
oMenuUI.WMENU10 = new sap.m.Menu({
    itemSelected: oAPP.events.ev_pressWmenuItemWS10,
    items: {
        path: "/WMENU/WS10/WMENU10",
        template: new sap.m.MenuItem({ key: "{key}", text: "{text}", icon: "{icon}", ... })
    }
}).addStyleClass("u4aWsWindowMenu");

// 3. 카테고리 버튼 리스트 (경로 바인딩)
new sap.m.Button({
    text: "{text}",
    press: oAPP.events.ev_pressWMenu10
}).bindProperty("visible", ...) // Test 메뉴는 R&D Staff만
```

### 메뉴 이벤트 분기 (fnHmws.js)

`ev_pressWmenuItemWS10` / `ev_pressWmenuItemWS20` 에서 선택된 메뉴의 `key` 값으로 분기한다.  
실제 동작은 `fnHmws.js` 에 정의된 `oAPP.fn.fnHmwsXX_XX_XX()` 함수들로 위임한다.

```js
// 예: WS10 Extras > App. Package Change
oAPP.fn.fnHmws10_10_10 = function(oEvent) { /* App. Package Change 팝업 */ };
// WS20 Edit > CSS Editor
oAPP.fn.fnHmws20_30_10 = function(oEvent) { oAPP.fn.fnEditorPopupOpener({OBJID:"STYLECSS1", OBJTY:"CS"...}) };
```

### 앱 검색 SearchField (id: AppNmInput)

```js
new sap.m.SearchField("AppNmInput", {
    value: "{/WS10/APPID}",             // JSONModel 양방향 바인딩
    change: ev_AppInputChange,           // 대문자 변환
    search: ev_AppValueHelp,            // F4 헬프 팝업 호출
    liveChange: function(oEvent) {       // 실시간 Suggestion 필터링
        oInput.getBinding("suggestionItems").filter([
            new sap.ui.model.Filter("APPID", FilterOperator.Contains, sValue.toUpperCase())
        ]);
        oInput.suggest(true);
    },
    enableSuggestions: true,
    suggestionItems: {
        path: "/WS10/APPSUGG",
        template: new sap.m.SuggestionItem({ key: "{APPID}", text: "{APPID}" })
    }
})
// 추가 이벤트: keydown (F4=115→fireSearch, Enter=13→fireChange, ESC/Home/End 처리)
//             dblclick (전체 선택)
//             mousedown (Suggestion 필터 갱신 및 suggest(true))
```

### 앱 검색 F4 헬프 팝업

`ev_AppValueHelp` → `oAPP.fn.fnAppF4PopupOpener(oOptions, fnCallback)` 를 호출한다.

```js
var oOptions = {
    autoSearch: true,
    initCond: {
        PACKG: "", APPID: sAppId, APPNM: "", APPTY: oAPP.attr.gAPPTY,
        EXPAGE: "WS10", ERUSR: sSapId, HITS: 500
    }
};
oAPP.fn.fnAppF4PopupOpener(oOptions, function(oAppData){
    APPCOMMON.fnSetModelProperty("/WS10/APPID", oAppData.APPID);
});
```

헬프 팝업 구현체는 `js/fnAppF4PopupOpen.js` / `js/moduleAppF4Popup.js` 에 분리되어 있으며, Electron BrowserWindow 팝업으로 열린다.

### 검색결과 처리 플로우

```
AppNmInput.fireChange(value)
  → ev_AppInputChange  : 값 대문자 변환
  → ev_AppDisplay / ev_AppChange
    → fnOnEnterDispChangeMode(APPID, ISEDIT)  [ws_fn_02.js]
      → 입력값 검증 (공백/특수문자)
      → fnCheckAppExists(APPID, callback)     [서버 호출: ajax_init_prc]
        → 성공: fnSetAppChangeMode() / fnSetAppDisplayMode()
                → NavContainer.to("WS20")
```

### 배경 컨텐츠 영역 (ws10_contPage)

`_getWs10ContentHtml()` 이 반환하는 인라인 HTML 마크업을 `sap.ui.core.HTML` 로 감싼다.  
테마에 따라 `u4a-ws-light-theme` CSS 클래스를 페이지에 추가/제거(`_applyWs10ContPageThemeClass`).  
`sap.ui.getCore().attachThemeChanged(...)` 으로 테마 변경 이벤트를 구독한다.

> **테마 컨버전** → [12_테마_컨버전_전략.md](12_테마_컨버전_전략.md) 따름. HTML5 컨버전 후 `u4a-ws-light-theme` 클래스 토글은 `document.documentElement.dataset.theme`(또는 `U4ATheme.apply(name)`) 전환으로, `attachThemeChanged` 구독은 `U4ATheme.onChange(cb)`(= `u4a-theme-changed` CustomEvent) 로 대체한다. 배경/글로우 색은 `theme/tokens.css` 의 의미 토큰만 소비하고 하드코딩하지 않는다(드리프트 금지).

## 7. WS20 디자인 영역 (Design Tree / Preview / Attribute)

> 대상 경로: `www/ws30/ws10_20/design/`

### 7.4 조회 / 편집 화면 진입 흐름

핵심 시작점은 `oAPP.fn.setUIAreaEditable` 이다.

**(A) 기존 데이터가 있고 Display 전환만 필요한 경우** — 다음 조건이면 서버 재조회 없이 Display 상태로 전환한다.

```javascript
IS_EDIT === "" && IS_CHAG === "" && APPDATA 존재 && isRefresh !== "X"
```

수행: `IS_EDIT=false` 설정 → Undo/Redo 초기화 → Tree D&D/Checkbox/Action Icon 활성 재계산 → Preview Drop 설정 제거 → CSS 미리보기 적용 제거 → Attribute 초기화 버튼 비활성화 → Attribute 라인 편집 비활성화 → 모델 refresh 후 Busy 해제.

**(B) 라이브러리 DB 정보가 이미 존재하는 경우** (`oAPP.DATA.LIB` 비어있지 않음) — Undo/Redo 초기화 → Design Layout 순서 설정 → `oAPP.fn.getAppData()` 호출.

**(C) 라이브러리 DB 정보가 없는 경우** — 화면 잠금 후 `T_9011, T_0020, T_0022, T_0023, T_0024, T_0027` 을 서버 조회. 완료 후:
1. `T_0022.UIOMD` 기준 `LIBNM` 구성 (예: `sap/m/Input` → `sap.m.Input`)
2. `T_9011` 공통코드 재구성
3. `UA035` 추가 정의
4. 어플리케이션 DB 정보 조회

### 7.14 Binding 흐름

Binding 아이콘(입력 필드 오른쪽 노란 아이콘) 클릭 → `oAPP.fn.attrIcon1Proc` → Property면 `attrBindProp`, Aggregation이면 `attrBindAggr`.

**Binding Popup 호출**: `oAPP.fn.callBindPopup(title, CARDI, callback, UIATK)`.

| `CARDI` | 의미 |
|---|---|
| `T` | TABLE만 가능 |
| `S` | STRUCTURE만 가능 |
| `F` | 일반 필드 가능 |
| `R` | RANGE TABLE만 가능 |
| `ST` | STRING_TABLE만 가능 |

Property binding 기본은 `F`. 예외: SelectOption2/3 value → `R`, Array property 중 숫자 아닌 타입 → `ST`. Aggregation binding 기본은 `T`.

**Binding 데이터 조회**: 서버 `/getBindAttrData` 응답의 `T_ATTR` 를 Tree 로 구성, 선택 가능 라인만 enable.

**Property Binding 처리** (`attrBindCallBackProp`): `is_attr.UIATV = selectedPath`, `is_attr.ISBND = "X"`, `is_attr.MPROP = additionalBindingProperty`, `is_attr.edit = false` → `attrSetBindProp` → `attrChange`. Binding된 Property는 직접 수정 불가. **Binding 해제 시**: `UIATV` 초기화 → default value 있으면 복원 → `ISBND` 초기화 → `MPROP` 초기화 → `attrChange` 재호출.

**Aggregation Binding 처리** (`attrBindCallBackAggr`): 기존 binding 또는 child binding 있으면 확인 팝업. 처리: Aggregation path 를 `UIATV` 설정 → `ISBND="X"` → UI의 `_MODEL`/`_BIND_AGGR` 에 binding context 구성 → 하위 UI model binding 관계 구성 → Tree parent/child 예외 속성은 필요 시 unbind.

### 7.15 Event 흐름

Event Attribute 는 서버 이벤트 목록을 DDLB 로 구성. 신규 서버 이벤트 생성은 DDLB 오른쪽 녹색 아이콘 → `oAPP.fn.attrCallEventPopup` → `oAPP.fn.createEventPopup` (`createEventPopup.js`).

### 7.17 디자인 영역 컨버전 / 수정 시 준수 원칙

디자인 영역은 SAP UI5 컨트롤(`sap.ui.table.TreeTable`, Preview iframe, Attribute Form)에 깊게 의존하므로, HTML5 컨버전 및 코드 수정 시 다음 동작 계약을 반드시 유지해야 한다.

1. **Design Tree 와 Preview 는 항상 동기화** — UI 추가/삭제/이동/복사 시 `zTREE`, `oAPP.attr.prev`, UI의 `_T_0015`, Preview iframe 내부 UI instance, Binding Popup Design Data, Undo/Redo history, Change Flag 를 동시에 갱신한다. Tree 만 또는 Preview 만 변경하지 않는다.
2. **UI 선택은 반드시 `oAPP.fn.setSelectTreeItem(OBJID)` 사용** — Tree 펼침, Row 선택, Attribute 갱신, Binding Popup 동기화, Preview 선택 표시, Attribute focus 를 함께 처리한다. 직접 row index 만 선택하지 않는다.
3. **Attribute 변경은 `attrChange → attrChangeProc → attrChgAttrVal` 흐름을 우회하지 않는다** — 입력값 검증, Undo history, Change Flag, `_T_0015` 수집, Preview 반영, Attribute style, Binding Popup 갱신, Preview 선택 유지가 포함된다. 직접 `_T_0015` 만 수정하지 않는다.
4. **UI 추가 가능 여부는 공통코드 기준** — `T_0022`, `T_0023`, `T_0027`, `T_9011`, `oAPP.attr.S_CODE` 기준으로 판단하고 `checkDenyChildAggr`, `checkAllowChildAggr`, `designChkFixedParentUI`, `chkUiCardinality` 검증을 우회하지 않는다. 하드코딩 금지.
5. **텍스트는 하드코딩하지 않는다** — 타이틀/버튼명/메시지/tooltip 은 접속 언어 기준 `oAPP.common.fnGetMsgClsText(...)` 또는 `parent.WSUTIL.getWsMsgClsTxt(...)` 를 사용한다. **(메시지 조회는 Node/REMOTE 기반 → 유지)**
6. **Preview iframe 내부 함수 호출 계약 유지** — `drawPreview`, `refreshPreview`, `selPreviewUI`, `createUIInstance`, `moveUIObjPreView`, `delUIObjPreView`, `destroyUIPreView`, `setChildUiException`, `setRichTextEditorException`, `removeDropConfig` 등의 파라미터 순서를 임의 변경하지 않는다.
7. **Rendering 대기 로직을 제거하지 않는다** — `parent.require(oAPP.oDesign.pathInfo.setOnAfterRender)` 를 통한 대기 로직. RichTextEditor, ObjectPage, Dialog, Page 등 예외 UI 는 rendering 타이밍이 중요하므로 단순 `refresh()` 로 대체하지 않는다.
8. **Undo / Redo history 누락 금지** — 주요 변경 action: `INSERT`, `DELETE`, `MULTI_DELETE`, `MOVE`, `DRAG_DROP`, `COPY`, `CHANGE_ATTR`, `RESET_ATTR`. Undo/Redo 호출 함수는 추가 파라미터가 있을 수 있으므로 signature 변경 시 호출부를 확인한다.
9. **Binding 처리된 속성은 직접 수정 가능 상태로 만들지 않는다** — `ISBND = "X"` Property 는 직접 입력 불가. 해제 시 default value 복원, `ISBND`/`MPROP` 초기화, DDLB binding item 제거를 함께 처리한다.
10. **삭제 시 하위 데이터까지 정리** — Client Event, Description, Binding 수집 정보, Popup 수집 정보, Preview instance, `oAPP.attr.prev[OBJID]`, `UA015UI` 예외 상태, 직접 입력 Aggregation 반영, 필수 Child UI 예외.

```text
이 흐름은 입력값 검증, Undo history, Change Flag, Preview 반영, Attribute style 처리, Binding Popup 갱신, Preview 선택 유지를 포함한다.

Binding은 attrBindProp 또는 attrBindAggr을 통해 callBindPopup을 호출한다.
Binding Popup은 /getBindAttrData 서버 응답의 T_ATTR를 기준으로 바인딩 가능한 필드를 표시한다.
Property Binding 시 선택 path를 UIATV에 매핑하고 ISBND = "X"로 설정하며, 직접 수정이 불가능해야 한다.
Binding 해제 시에는 default value 복원, ISBND 초기화, MPROP 초기화, DDLB binding item 제거를 함께 처리한다.

Event Attribute에서 신규 서버 이벤트를 만들 때는 createEventPopup을 사용한다.
이벤트명은 대문자로 변환하고 EV_ prefix가 없으면 자동으로 추가한다.
서버 생성 후 attrCreateEventCallBack에서 이벤트명을 Attribute 값으로 매핑하고 attrChange 흐름을 수행한다.
Event는 UI에 직접 attach하지 않고 저장용 Attribute 정보로 수집한다.

ROOT 선택 시에는 일반 UI Attribute가 아니라 T_9011의 UA003 기준으로 Document Attribute를 구성한다.
ROOT Attribute 변경 중 일부는 appInfo에도 반영된다.

모든 화면 텍스트, 버튼명, 메시지, tooltip은 접속 언어 기준 텍스트 함수인 oAPP.common.fnGetMsgClsText 또는 parent.WSUTIL.getWsMsgClsTxt를 사용해야 한다.
하드코딩된 한글/영문 텍스트를 추가하지 않는다.

코드 수정 시 기존 Busy 처리, Shortcut Lock 처리, Undo/Redo 처리, Binding Popup 동기화, Preview rendering 대기 로직을 제거하지 않는다.
특히 RichTextEditor, Dialog, Page, ObjectPage 계열 UI는 rendering 타이밍 예외가 있으므로 setOnAfterRender 모듈을 통한 대기 로직을 유지한다.
```

**Node FS 의존 → 유지**

### Footer Message 시스템

**단건 메시지 (fnShowFloatingFooterMsg)**

- 모델 `/FMSG/WS10` 또는 `/FMSG/WS20` 의 `{ISSHOW, ICON, ICONCOLOR, TXT}` 를 업데이트
- `sap.m.Page.showFooter` 가 `ISSHOW` 에 바인딩되어 자동 표시/숨김
- 10초 후 자동 숨김 (`setTimeout`)
- 사운드 출력: `parent.setSoundMsg('01' | '02')` **유지**

**멀티 메시지 (fnShowMultiFooterMsg)**

- `fnShowMultiFooterMsg.js` 의 코드는 전체 주석 처리됨 (폐기)
- 현행은 `ERRMSGPOP` Electron 팝업 창으로 대체
- `APPCOMMON.fnMultiFooterMsgClose()` → `REMOTE.getCurrentWindow().getChildWindows()` 에서 `ERRMSGPOP` 창 찾아 `close()`

### 12.3 기능 동등성 — 기능 인벤토리 (전수 보존)

> 이 절은 **설계 산출물**이다. 셸 크롬 컨버전의 제1원칙은 **기능 동등성(Functional Parity)** 이다.

**원칙** — 기존 기능·동작을 **하나도 빠짐없이 동등하게** 재현한다. 판정 기준은 외형(마크업이 비슷한가)이 아니라 **행동**(같은 입력에 같은 결과·부작용·상태 전이가 나오는가)이다. UI5 컨트롤을 HTML 요소로 바꾸는 과정에서 임의 단순화·생략이 발생하기 쉬우므로, 변환 착수 전 아래 카테고리별로 **현행 동작을 전수 목록화**하고, 항목마다 HTML5 구현 위치와 검증 방법을 1:1로 채운다.

### 12.4 작업 순서 (각 단계 = 입력 → 산출 → 수용 기준)

1. **Electron/Node 자원 보존 확인** — 11장 (A) 의 자원(`parent.REMOTE`, `IPCRENDERER/IPCMAIN`, `parent.FS`, `parent.CURRWIN`, `parent.setBusy`, `sendAjax`, `parent.WSUTIL`, `BroadcastChannel`, `better-sqlite3` 등)은 **호출부를 그대로 유지**한다. UI 레이어만 교체. 수용: 11장 (A) 항목 코드 변경 0건.
2. **셸 마크업 교체** — 11장 (B) 매핑대로 `sap.m.App`/`NavContainer`/`Page`/`Bar`/`OverflowToolbar`/`ToolPage`/`SideNavigation`/`Menu` 등을 HTML 요소로. 페이지 컨테이너 전환은 `NavContainer.to(id)` → `showPage(id)`(11장 B), 헤더 윈도우 메뉴/SideNav 는 `renderMenu`/`<ul><li>` 로. 수용: 셸에서 UI5 컨트롤(`sap.m.*`/`sap.tnt.*`) 미생성, `.codex_tmp/ws10_html5` 시안과 구조 일치.
3. **상태/렌더 분리** — Core `JSONModel`(`/WS10`, `/WS20`, `/WMENU`, `/FMSG`) → 단일 상태객체(`WS_STATE`) + `setState/getState`(11장 말미 요약), `bindProperty`/리스트 바인딩 → `render*()` 함수. 수용: 모델 변경 시 DOM 동기화(버튼 visible, 메뉴, 푸터 메시지, APPID/모드/활성 타이틀) 동작.
4. **테마 토큰 소비** — 셸의 모든 색·radius·shadow 를 `theme/tokens.css` 의미 토큰(12번 6.2)으로. WS10 배경/글로우(5장)·헤더·툴바·트리·Attribute 패널 포함. 수용: `shell.css`·셸 CSS 내 hex 색상·하드코딩 radius·shadow **0건**(grep 검증) **+ 12번 6.6 체크리스트 표면**(스크롤바·윈도우 메뉴/SideNav 드롭다운 펼침·focus 링·hover/disabled·ValueState·placeholder·오버레이·타이틀바 버튼 hover 등) 전수 토큰화.
5. **테마 전환 연동** — `applyTheme`/`u4a-ws-light-theme` 클래스 토글 → `U4ATheme.apply(name)`(12번 5.1), `attachThemeChanged` → `U4ATheme.onChange(cb)`(12번 5.2). sap_horizon 기준 5종(white/dark/purple/red/green), 기본 white. 수용: 테마 전환 시 셸 크롬 색상 함께 변경, Preview iframe 은 영향 없음.
6. **텍스트 비하드코딩** — 타이틀/버튼명/메시지/tooltip 은 접속 언어 기준 `oAPP.common.fnGetMsgClsText` / `parent.WSUTIL.getWsMsgClsTxt`(7.17, 9장). 수용: 셸 문구 하드코딩 없음.
7. **디자인 영역 경계 확인** — WS20 메인의 Design Tree/Preview/Attribute 는 본 절에서 교체하지 않고 **7장·7.18 계약을 유지**(setSelectTreeItem, attrChange 흐름, Preview iframe 함수 시그니처, UI5 테마). 수용: 셸 컨버전이 디자인 영역 동작 계약을 깨지 않음.

### 12.5 불변 제약 (반드시 보존)

- **기능 동등성** — 기존 기능·이벤트·단축키·검증·상태·서버분기·엣지케이스를 하나도 빠짐없이 동등하게 재현(행동 기준). 임의 단순화·생략 금지(12.3).
- **Electron/Node 자원 전부 유지** — 11장 (A). 특히 `parent.CURRWIN`(minimize/maximize/close/setTitle/flashFrame), `parent.setBusy`, `sendAjax`, IPC 채널(`if-*`), `BroadcastChannel`, `parent.WSUTIL`(SQLite/레지스트리/FS).
- **Activate/Save 서버 흐름 불변** — 6장의 `ev_pressActivateBtn`/`ev_pressSaveBtn`: `getSaveData()` → `_convertSaveDataToFormData()` → `sendAjax("/save_active_appdata#active|#save")` → `RETCD` 분기 → IPC `command("activate")` 전파. `fnSetBusyLock` Lock/Busy 쌍, `fnMoveToWs10`/`fnSetAppDisplayMode` 분기. UI만 교체하고 흐름은 건드리지 않는다.
- **Preview iframe 내부 UI5 테마 유지** — 7장. 셸 테마 토큰을 iframe 안으로 침투시키지 않는다(12번 1.1, 6.4).
- **테마는 12번 문서 단일 출처** — 셸 색/모양은 토큰만 소비, 하드코딩·드리프트 금지(12번 5장·6.4).
- **메뉴 key 분기 계약 유지** — 윈도우 메뉴 항목 동작은 `fnHmws[페이지][카테고리][항목]`(9장) key 문자열 분기. 메뉴 `key`(`WMENU10`… / `M03`·`M04`·`M05`) 와 `itemSelected`→`fnHmwsXX_XX_XX` 위임 계약, 메뉴 데이터 모델 경로(`/WMENU/WS10` 등)를 바꾸지 않는다.
- **페이지 전환 계약 유지** — `oApp.to("WS10"|"WS20")`/`back()` 의미를 `showPage(id)` 로 보존하고, 전환 시 `parent.setCurrPage(toId)` 통지(1장)·단축키 set/remove(`setShortCut`/`removeShortCut`)·세션 정리(`fnKillUserSession`)·Lock 해제를 누락하지 않는다.
