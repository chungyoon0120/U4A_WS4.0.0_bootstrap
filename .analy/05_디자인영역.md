## 1. 개요 — 전체 구성 및 IDE UI vs 대상앱 미리보기 구분

### 1-1. 3분할 레이아웃

디자인 영역은 WS10_20 컨텍스트에서 단일 SPA처럼 동작하며, 화면을 다음 3개 페이지(Panel)로 구성한다.

| 영역 | 코드상 명칭 | 담당 파일 | 주 UI5 컨트롤 |
|------|------------|-----------|--------------|
| 좌측 — UI 디자인 트리 | `oLTree1`, `oLPage` | `js/uiDesignArea.js` | `sap.ui.table.TreeTable` |
| 중앙 — 미리보기 | `prevHTML` iframe, `oMPage` | `js/uiPreviewArea.js` + `preview/index.js` | `sap.ui.core.HTML`(iframe 래퍼) |
| 우측 — 속성 편집 | `oRPage` | `js/uiAttributeArea.js` + `attributesArea/` | `sap.f.DynamicPage`, `sap.m.List` |

전체 진입점은 `js/main.js`(`oAPP.fn.main`)로, 광역 상태(`oAPP.attr`, `oAPP.oDesign`)를 초기화한 뒤 각 영역 생성 함수를 순서대로 호출한다.

### 1-2. IDE UI vs 대상앱 미리보기 — 핵심 구분

```
┌─ Electron BrowserWindow (ws10_20 페이지) ─────────────────────┐
│  [IDE 자체 UI — SAP UI5로 구성]                                │
│  ┌── 트리 영역 ──┬── 미리보기 iframe ──┬── 속성 영역 ──┐      │
│  │ sap.ui.table  │  <iframe id=prevHTML>│ sap.f.Dynamic │      │
│  │  .TreeTable   │  ┌─────────────────┐│ Page +        │      │
│  │               │  │ [대상앱 미리보기]││ sap.m.List    │      │
│  │               │  │ UI5 sap-ui-core │││               │      │
│  │               │  │ 가 iframe 안에  │││               │      │
│  │               │  │ 로드되어 대상앱  │││               │      │
│  │               │  │ UI5 컨트롤을    │││               │      │
│  │               │  │ 동적 렌더링함   │││               │      │
│  └───────────────┴──└─────────────────┘┴───────────────┘      │
└────────────────────────────────────────────────────────────────┘
```

- **IDE 자체 UI (컨버전 대상)**: 트리, 속성패널, 툴바, 팝업 등 — 모두 `sap.*` 컨트롤로 구성됨.
- **미리보기 영역 (별도 판단 필요)**: `<iframe id="prevHTML">`이 `preview/index.js`를 로드. 이 iframe 내부에 SAP UI5 부트스트랩(`sap-ui-core.js`)이 별도 실행되어 사용자가 설계한 대상앱 UI5 컨트롤을 **동적으로 인스턴스화·렌더링**한다. 즉 미리보기의 `sap.*` 컨트롤은 **사용자 대상앱의 UI5**이므로, 컨버전 대상에서 분리하여 별도 판단해야 한다.

### 2-2. UI 노드 구조 — T_0014(ZTU4A0014) 레코드

`js/main.js`의 `oAPP.fn.crtStru0014`에서 정의:

```js
{
  APPID, GUINR,         // 앱 ID, GUI 번호
  OBJID,                // UI 노드 고유 ID (예: "ROOT", "BTN001")
  POSIT,                // 위치(순번)
  POBID,                // 부모 OBJID
  UIOBK,                // UI Object Key (예: "UO00100" = sap.m.Button)
  PUIOK,                // 부모 UI Object Key
  ISAGR,                // Aggregation 여부
  AGRID,                // Aggregation ID
  ISDFT,                // Default Aggregation 여부
  OBDEC,                // 설명
  AGTYP,                // Aggregation 타입
  UIATK,                // UI Attribute Key (부모 aggregation)
  UIATT,                // UI Attribute 명 (aggregation 명)
  UIASN,                // Attribute Simple Name
  UIATY,                // Attribute 유형 (1:property, 2:event, 3:aggregation, 4:association, 6:embed)
  UIADT,                // Attribute Data Type
  UIADS,                // Attribute Description
  VALKY,                // Value Key
  ISLST, ISMLB,         // 리스트 여부, 다건 입력 여부
  TOOLB, UIFND,         // Toolbar 여부, UI 분류 코드
  UILIB,                // 라이브러리명 (예: "sap.m.Button")
  ISEXT,                // Extension UI 여부
  TGLIB,                // Target Library
  ISECP                 // 예외처리 UI 여부
}
```

트리는 `POSIT(순번)`와 `POBID(부모 ID)` 필드로 부모-자식 관계를 표현하며, `js/main.js`의 `lf_setTreeData` 함수가 평면 배열을 재귀 트리(`zTREE`)로 변환한다.

### 2-3. 트리 조작

- **추가**: 컨텍스트 메뉴(M01) 또는 단축키 → `callUIInsertPopup` → T_0014 레코드 신규 생성 후 모델 반영
- **삭제**: 컨텍스트 메뉴(M02) → Undo 이력 저장("DELETE") → 트리에서 제거
- **이동**: D&D(`sap.ui.core.dnd.DragInfo/DropInfo`) 또는 `uiMovePosition` 팝업
- **검색**: `callDesignTreeFindPopup` — keyword 입력 → `Indication03`/`Indication07` 하이라이트
- **필터**: `callDesignTreeFilterPopup` — `designSetFilter(value)` 호출

## 4. 미리보기 렌더링

### 4-1. 아키텍처 — UI5 동적 렌더링 방식

```
[IDE BrowserWindow]
   oAPP.fn.loadPreviewFrame()
       → <iframe id="prevHTML"> src = design/preview/index.html
             ↓ (iframe 내부 별도 JS 컨텍스트)
   [preview/index.js]
       1. sap-ui-core.js 부트스트랩 (UI5 전체 프레임워크 로드)
       2. definePreviewControl() → sap.ui.define("u4a.m.Preview", ...)
          커스텀 Control 정의 (content aggregation 지원)
       3. drawPreview() 호출
          ├→ setUIScript(oAPP.attr.oModel.oData.zTREE)
          │   └→ createUIInstance(is_tree, it_0015) — 재귀
          │       ├→ sap.ui.requireSync(LIBNM) — UI5 라이브러리 동기 로드
          │       ├→ new [SapUI5Class](uid, {property:value, ...})
          │       └→ oAPP.attr.prev[OBJID] = UI5 인스턴스
          ├→ setUIParent() — addAggregation으로 부모-자식 연결
          └→ prevRootPage.placeAt("Content") — DOM 마운트
```

**핵심**: 미리보기 iframe은 사용자가 설계한 SAP UI5 컨트롤을 **실제 UI5 런타임으로 인스턴스화**하여 보여준다. 단순 HTML 렌더링이 아니라 완전한 UI5 컨트롤 트리를 DOM에 플레이스한다.

### 4-2. IDE ↔ preview iframe 통신

```
// IDE(parent)에서 preview(iframe contentWindow) 직접 호출
oFrame.contentWindow.drawPreview()
oFrame.contentWindow.setPreviewZoom(value)
oFrame.contentWindow.setUiLoadLibraries(it_lib)
oFrame.contentWindow.getIconList()

// preview에서 IDE(parent) 접근
parent.oAPP.attr.prev[OBJID]   // UI5 인스턴스
parent.oAPP.DATA.APPDATA       // 앱 데이터
parent.oAPP.fn.*               // IDE 기능 함수
u4aRootParent.require(path)    // Electron Node.js require (parent.parent)
```

### 4-3. setAfterRendering / setOnAfterRender

두 파일이 존재하며 역할이 유사하나 호출 컨텍스트가 다르다:

| 파일 | 위치 | 용도 |
|------|------|------|
| `js/setAfterRendering.js` | IDE 컨텍스트(CommonJS `module.exports`) | 팝업류 UI의 `onAfterRendering` → `attachAfterOpen` 대기 |
| `js/previewRender/setOnAfterRender.js` | `u4aRootParent.require`로 preview에서 호출 | `onAfterRendering` + `awaitPreviewExceptionUI` (am5 차트, RichTextEditor 등 비동기 완료 대기) |

두 모듈 모두 `sap.ui.base.ManagedObject.prototype.invalidate` 패턴을 감지해 실제 렌더링 책임 UI를 찾아 올라가는 `getTargetAfterRenderingUI` 패턴을 사용한다.

### 4-4. 예외 UI 처리

- **RichTextEditor**: `readyRecurring` 이벤트 대기 (`renderingRichTextEditor.js`)
- **sap.m.BusyDialog**: `_oDialog` 내부 프로퍼티로 실제 대화상자 참조
- **sap.m.QuickView**: `_oPopover`
- **sap.m.Menu**: `_getMenu()` 호출로 내부 메뉴 획득
- **am5 차트 등**: `awaitPreviewExcepionUI` — 공통코드 UA015 기반 추가 대기
- **sap.m.TileContainer**: `preview/index.js` 내 `lf_excepRequire`에서 prototype 메서드 오버라이드

### 4-5. Skeleton Screen

`js/prevSetSkeletonScreen.js` — `sap.m.Dialog`로 구성된 팝업에서 현재 미리보기 레이아웃 기반으로 Skeleton Screen 설정 정보를 생성·저장.

## 6. 바인딩 & 이벤트

### 6-1. 바인딩 팝업 (`js/callBindPopup.js`)

`oAPP.fn.callBindPopup(sTitle, CARDI, f_callback, UIATK)`:
- `CARDI` 값에 따라 바인딩 대상 타입 제한 (T:테이블, S:구조체, F:필드, R:RANGE TABLE, ST:STRING_TABLE)
- `oAPP.attr.oBindDialog` — 재사용 dialog 인스턴스
- 바인딩 선택 결과 → `f_callback`으로 T_0015 ISBND, BPATH 갱신

### 6-2. BroadcastChannel 통신 (`bindPopupHandler/broadcastChannelBindPopup.js`)

디자인 화면(WS20)과 바인딩 팝업(별도 Electron 창) 간 실시간 통신:

```js
const C_CHID = `${sBrowserKey}_ws20_bindpop`;
oChannel = new BroadcastChannel(C_CHID);
```

처리 코드(`PRCCD`) 목록:

| PRCCD | 방향 | 내용 |
|-------|------|------|
| `BUSY_ON` | 바인딩팝업 → WS20 | Busy 활성 요청 |
| `BUSY_OFF` | 바인딩팝업 → WS20 | Busy 해제 요청 |
| `UPDATE-DESIGN-DATA` | 바인딩팝업 → WS20 | 앱 데이터(T_0015) 갱신 |
| `ROOT_OBJID` | 바인딩팝업 → WS20 | 미리보기 최상위 OBJID 전달 |

D&D 시 `bindPopupHandler/setDragBindPopupData.js`가 `event.dataTransfer.setData("prc002", JSON.stringify(_sParam))`으로 바인딩 팝업에 트리 데이터 전달.

### 6-3. 이벤트 생성 팝업 (`js/createEventPopup.js`)

`oAPP.fn.createEventPopup(is_attr, f_callBack)`:
- 이벤트 메서드명 입력 (필수: `EV_` prefix 자동 부여)
- Description 입력
- Trial 버전 차단 로직 포함
- 완료 후 T_0015의 UIATY=2 레코드에 메서드명 저장

### 6-4. 단축키 자동 등록 (`attrContextMenu/eventShortcutReg.js`)

`shortcutList.js`에 정의된 허용 가능 이벤트 목록(예: `sap.m.Button.press`, `sap.f.ShellBar.menuButtonPressed` 등 ~60개)과 현재 attr을 대조하여, 서버이벤트가 등록된 경우 단축키 자동 등록 처리.

## 8. Undo/Redo — 변경 데이터 마킹/필터

### 8-1. Undo/Redo 아키텍처 (`undoRedo/undoRedo.js`)

## 10. setAppScript — 앱 스크립트 처리

**파일**: `js/setAppScript.js`  
**함수**: `oAPP.fn.setAppScript(it_0014, it_0015)`

T_0014(UI 트리) + T_0015(attr) 데이터를 기반으로 **UI5 JavaScript 코드 문자열을 생성**하는 모듈.

처리 흐름:
1. `lf_setTreeData(it_data, parent, child)` — 평면 배열 → 재귀 트리
2. `lf_setProp(OBJID)` — property(UIATY=1) 항목을 `key:"value"` 형식으로 조립
3. `lf_setEvent(OBJID)` — event(UIATY=2) 항목을 `attachXxx(oAPP.fn.EV_XXX)` 형식으로 조립
4. `lf_setAggr(OBJID)` — aggregation(UIATY=3) 항목을 `addXxx(child)` 형식으로 조립
5. SKIP 조건: ROOT(`OBJID === "ROOT"`), EXTENSION UI(`ISEXT==="X"`), EXCEPTION UI(`ISECP==="X"`)
6. 데이터 타입(`UIADT`)에 따라 숫자/boolean은 따옴표 없이, 문자열은 쌍따옴표로 감싸 처리

결과 스크립트는 컨트롤러 파일이나 미리보기 렌더링 전 T_0014/T_0015 → JS 코드 변환용으로 활용된다.

#### B-2. 미리보기 영역의 UI5 동적 렌더링 — 가장 어려운 핵심 문제

**현재 방식**:  
`preview/index.js`가 iframe 내에서 `sap-ui-core.js`를 완전 부트스트랩하고, 사용자 설계 데이터(T_0014/T_0015)를 바탕으로 SAP UI5 컨트롤을 `new sap.m.Button(...)` 등으로 인스턴스화하여 DOM에 렌더링한다. 이것은 **대상앱이 UI5 앱**이기 때문에 발생하는 구조이며, IDE 자체와 분리되어 있다.

**컨버전 시 선택 가능한 전략**:

| 전략 | 설명 | 난이도 | 권장 여부 |
|------|------|--------|----------|
| **① 현상 유지 (권장)** | 미리보기 iframe 내 UI5 렌더링은 그대로 유지. IDE 자체 UI(트리/속성/툴바)만 바닐라 HTML로 교체. iframe 경계를 유지하면 두 레이어가 격리됨. | 낮음 | **강력 권장** |
| ② iframe을 별도 Electron WebView로 격리 | `<webview>` 태그 사용으로 UI5 런타임을 완전 격리. 통신은 IPC로만 | 중간 | 선택적 |
| ③ 미리보기도 HTML 렌더러로 교체 | T_0014/T_0015를 파싱해 순수 HTML 생성. 단, UI5 컨트롤의 동작(이벤트, 바인딩, 다이나믹 패널 등)을 완전 재현 불가 | 매우 높음 | **비권장** |

**전략 ①의 구체 방안 (경계 인터페이스 정리)**:

```
[IDE — 바닐라 JS 리팩토링 후]
  ↓ iframe 통신 인터페이스는 현행 유지
  oFrame.contentWindow.drawPreview()
  oFrame.contentWindow.setUiLoadLibraries(it_lib)
  oFrame.contentWindow.setPreviewZoom(value)

[preview/index.js — 변경 없음]
  ↓ parent.oAPP로 IDE 상태 접근 (현행 유지)
  parent.oAPP.attr.prev[OBJID] = UI5 인스턴스
  parent.oAPP.DATA.APPDATA.T_0015 (현행 유지)
```

**이 경우 주의사항**:
- IDE 자체 `sap.*` 객체(트리, 속성 패널 등)가 제거되면 `parent.sap`이 undefined가 됨. preview/index.js는 자체 iframe 내 `sap`을 사용하므로 직접 영향 없음. 단, `preview/index.js`에서 `parent.oAPP.fn.*`, `parent.oAPP.attr.*`을 광범위하게 참조하므로 IDE 리팩토링 시 해당 API 시그니처는 반드시 유지해야 함.
- `callDesignContextMenu`는 preview 내에서 `this.sap.m.Menu`로 IDE 컨텍스트의 sap 객체를 사용함. IDE UI5 제거 후에는 이 컨텍스트 메뉴도 바닐라 HTML로 교체하거나, preview 자체 sap으로 전환해야 함.
- `sap.ui.core.dnd.DragInfo/DropInfo`를 preview 내에서 사용하므로 D&D 이벤트 처리는 preview 내 UI5 DnD API를 그대로 사용.

### 13.3 작업 순서 (각 단계 = 입력 → 산출 → 수용 기준)

1. **Electron/Node 자원 보존 확인** — 12장 표 A 의 자원(`REMOTE`/`BroadcastChannel`/`parent.require`/`u4aRootParent.require`/`better-sqlite3`/Electron IPC/Worker)은 **호출부를 그대로 유지**. UI 레이어만 교체. 수용: 표 A 항목 코드 변경 0건.
2. **UI 마크업 교체** — 12장 표 B-1 매핑대로 IDE 호스트 UI5 컨트롤(`sap.ui.table.TreeTable`/`sap.f.DynamicPage`/`sap.m.List`/`sap.m.Dialog`/`sap.m.Menu` 등) → HTML 요소. 수용: IDE 화면에서 `sap.*` 컨트롤 미생성, 기존 UI5 디자인 영역 구조(좌측 트리·중앙 미리보기·우측 속성)와 일치.
3. **모델/zTREE/_T_0015 → 상태·렌더 분리** — `JSONModel` 바인딩 → JS 상태객체 + `render*()` 함수. `zTREE`(트리), `_T_0015`(속성 수집), `oAPP.attr.prev`(미리보기 인스턴스 참조)는 **동기화 단위로 함께 갱신**(03 7.17-1). 수용: 모델 변경 시 DOM 동기화 동작, Tree만/Preview만 변경되지 않음.
4. **테마 토큰 소비** — 모든 색·radius·shadow 를 `theme/tokens.css` 의 의미 토큰(12번 6.2)으로. `changedDataMarker` 의 `changeValue` 등 상태 표시 CSS 도 토큰 기반으로. 수용: `design/css/common.css` 및 컴포넌트 CSS 내 hex 색상·하드코딩 radius·shadow **0건**(grep 검증) **+ 12번 6.6 체크리스트 표면**(스크롤바·트리 expand/indent guide·row hover/선택·드롭다운 펼침·focus 링·hover/disabled·ValueState·드래그 드롭 타깃·placeholder·오버레이 등) 전수 토큰화(미리보기 iframe 제외).
5. **테마 전환 연동** — `sap.ui.getCore().applyTheme(...)` 대신 `U4ATheme.apply(name)`(12번 5.2). `data-theme` 속성 전환으로 sap_horizon 기준 5종 테마(white/dark/purple/red/green) 적용(기본 white). 수용: 테마 전환 시 셸(트리·속성·툴바) 색상이 함께 변경되고(모양은 horizon 공통 유지), **미리보기 iframe 내부는 영향받지 않음**.
6. **반응형 레이아웃** — 3분할(트리·미리보기·속성)을 `%`/`fr`/`clamp()` + Grid 로 유동 구성하고(고정 px 폭 금지), 스플리터 드래그(`--tree-*`/`--attr-*` min/basis/max)·브레이크포인트 전환(L 3분할 → M 속성 토글 2분할 → S 1패널+탭)·OverflowToolbar·container query·터치 적응을 **12번 7장**대로 구현한다. UI5 가 제공하던 반응형(OverflowToolbar/ResponsivePopover/DynamicPage/Splitter)을 직접 재현한다. **미리보기 iframe 내부 반응형은 UI5 가 처리하므로 침범 금지**(셸 폭만 조정, iframe 통신 시그니처 불변). 수용: `design/css/common.css` 에 고정 px 폭 **0건**(grep), 창 리사이즈·스플리터 양끝 드래그·S/M/L/XL × 터치(`pointer:coarse`)·400% 확대에서 겹침·잘림·가로 스크롤 폭주 없음. 12번 6.6 체크리스트 **J(반응형)** 전수 통과.
7. **텍스트 비하드코딩** — 타이틀/버튼명/메시지/tooltip 은 접속 언어 기준 `oAPP.common.fnGetMsgClsText` / `parent.WSUTIL.getWsMsgClsTxt`(03 7.17-5). 수용: 화면 문구·메시지키 하드코딩 없음.

### 13.4 불변 제약 (반드시 보존)

- **Preview iframe 내부는 실제 UI5 → UI5 테마 유지** — 미리보기(`preview/index.js`)는 사용자 설계 대상앱을 실제 UI5 런타임으로 렌더하므로 변환 대상이 아니다(1-2절, 12장 표 B-2 전략①). 셸 토큰/`data-theme`/`tokens.css` 를 **iframe 안으로 침투시키지 않는다**(12번 6.4, 1.1). iframe 통신 인터페이스(`drawPreview`/`setUiLoadLibraries`/`setPreviewZoom` 등)와 `parent.oAPP.*` 시그니처는 그대로 유지한다.
- **Design Tree ↔ Preview 동기화** — UI 추가/삭제/이동/복사 시 `zTREE`·`oAPP.attr.prev`·`_T_0015`·Preview iframe 인스턴스·Binding Popup·Undo/Redo·Change Flag 를 동시 갱신한다(03 7.17-1). Tree만/Preview만 변경 금지.
- **`setSelectTreeItem` / `attrChange` 흐름 우회 금지** — UI 선택은 `oAPP.fn.setSelectTreeItem(OBJID)` 로(03 7.17-2), 속성 변경은 `attrChange → attrChangeProc → attrChgAttrVal` 로(03 7.17-3). 직접 row index 선택·직접 `_T_0015` 수정 금지. Preview 함수 파라미터 순서·`setOnAfterRender` 렌더 대기 로직 유지(03 7.17-6·7).
- **테마는 12번 문서 단일 출처** — 셸의 색은 토큰만 소비, 하드코딩·드리프트 금지. `--accent` 등 동일 의도 색은 단일 토큰(12번 6.4). 모양(radius/elevation/spacing)은 **sap_horizon 공통 고정**이며 테마는 색만 바꾼다(12번 2.1).
- **반응형 필수** — 3분할 셸을 고정 px 폭으로 못 박지 않는다. 유동 단위·container query·12번 4.3 브레이크포인트(600/1024/1440)·12번 4.3 레이아웃 토큰(`--tree-*`/`--attr-*`/`--space-*`/`--hit-area`)만 소비해 창 리사이즈·스플리터·터치에 대응한다(12번 7장). UI5 반응형(OverflowToolbar/ResponsivePopover/DynamicPage/Splitter)은 직접 재현하되, **미리보기 iframe 내부 반응형은 UI5 영역(KEEP-UI5)** 이므로 침범하지 않는다.
- **Electron/Node 자원 전부 유지** — 12장 표 A. `BroadcastChannel`(WS20↔바인딩팝업), `attrPresetPopup` 별도 `BrowserWindow`, `better-sqlite3`, `u4aRootParent.require` 등 호출부 불변.
- **기능 동등성** — 기존 기능·이벤트·단축키·검증·상태·서버분기·엣지케이스를 빠짐없이 동등 재현(행동 기준). 03 7.17 동작 계약 준수. 임의 단순화·생략 금지.
