## 1. 3대 전략 (반드시 채택)

### 전략 A — "override" 방식 (원본 무수정)
- 원본 UI5 JS(`ws_fn_01.js` 4천줄, `uiDesignArea.js` 6.8천줄 등)를 **직접 고치지 않는다.**
- 부팅 시 원본을 그대로 로드한 뒤, **맨 마지막에 `*_html5_*.js`(신규 파일)를 로드해 같은 이름의 `oAPP.fn.*` 함수를 재정의(override)** 한다.
- 효과: 데이터/비-UI 함수는 원본 그대로 호출 + UI 함수만 교체. 롤백/비교 용이.
- **철칙**: override 파일은 **재정의 대상보다 반드시 뒤에 로드**(로드 순서가 곧 우선순위).

### 전략 B — 미리보기는 UI5 유지 (KEEP-UI5)
- 미리보기 iframe(`design/preview/`)은 **수정 금지**. 자체적으로 SAP 서버에서 UI5 부트스트랩을 받아 대상 앱을 렌더.
- 변환은 iframe을 삽입하는 **IDE 호스트 + Preview 툴바**만.

### 전략 C — UI5 모델 → "경로기반 상태객체 shim"
- 원본 `sap.ui.model.json.JSONModel` → `getProperty("/A/B")` / `setProperty("/A/B",v)` / `setData` / `getData` / `refresh()` / **`oData` getter**(내부 데이터 직접 노출)를 가진 순수 객체로 대체.
- 바인딩(`{/path}`) → 상태객체 갱신 + 수동 DOM 재렌더 함수.

## 2. ★ 시작하자마자 깔아야 할 "기반 보정" 4종

> 화면 변환 전에 **이 4개를 먼저** 적용하지 않으면, 변환 안 한 화면의 엣지/에러 경로에서 반드시 터진다. 시행착오로 가장 비쌌던 부분.

### 2-1. 전역 `sap` 위임 (메인프레임 → 미리보기 iframe)
- **왜**: 원본 IDE는 UI5 기반이라 전역 `sap`이 존재했다. UI5 제거 시 사라지는데, **복사된 IDE 코드 곳곳이 bare `sap`을 참조**(예: `setOnAfterRender.js` `getTargetAfterRenderingUI`, `ws_fn_02.js` `fnCloseAllDialog` 등) → `ReferenceError: sap is not defined`. 이 코드들이 다루는 UI5 컨트롤은 미리보기 iframe에 살아있다.
- **해결**: 전역 진입 스크립트(예: 메인프레임 공통 `resources/index.js`급)에서 메인프레임 전역 `sap`을 **미리보기 iframe의 `sap`으로 지연 위임**.
  ```js
  (function () {
    if (typeof window.sap !== "undefined") return;
    var _override;
    function getSap() {
      if (typeof _override !== "undefined") return _override;
      try { var f = document.getElementById("prevHTML"); return (f && f.contentWindow && f.contentWindow.sap) || undefined; }
      catch (e) { return undefined; }
    }
    var d = { configurable: true, get: getSap, set: function (v) { _override = v; } };
    try { Object.defineProperty(window, "sap", d); } catch (e) {}
    // Node require 모듈(예: setOnAfterRender.js)의 bare sap 은 global 스코프로 해석되므로 양쪽 정의
    try { if (typeof global !== "undefined" && global !== window && typeof global.sap === "undefined") Object.defineProperty(global, "sap", d); } catch (e) {}
  })();
  ```
- iframe 미로드 시 `undefined` 반환 → `typeof sap === "undefined"` / `sap?.x` 가드가 그대로 동작. **`window` 와 `global` 양쪽** 정의 필수.

### 2-2. 전역 `jQuery` 복원
- **왜**: 원본은 UI5 부트스트랩이 전역 jQuery를 제공. 게다가 Electron `nodeIntegration`에선 `<script src="jquery">`가 jQuery를 `window` 아닌 `module.exports`로 등록 → 전역이 빔.
- **해결**: 각 HTML 진입 파일에서 jQuery 로드 시 `module`을 잠깐 숨긴다.
  ```html
  <script>window.__m=window.module; window.module=undefined;</script>
  <script src="<경로>/jquery.min.js"></script>
  <script>window.module=window.__m; delete window.__m;</script>
  ```

### 2-3. `showCriticalErrorDialog` 전역 wrapper (APP 종료 방지)
- **왜**: 미리보기 iframe의 `window.onerror`는 `parent.parent.showCriticalErrorDialog(msg)`를 호출. 원본 `ws_trycatch` 버전은 **`APP.exit()`** 까지 함 → 미리보기 1회 오류에 앱 전체 종료.
- **해결**: 메인프레임 전역에 Electron 네이티브 다이얼로그 **표시만 하고 종료 안 하는** wrapper 추가.

### 2-4. 모델 shim에 `oData` getter + `refresh()`
- 원본 디자인 코드가 `oModel.oData.zTREE`(직접 접근)와 `oModel.refresh()`를 광범위하게 호출. shim에 **`oData` getter**(내부 데이터 노출)와 **`refresh()`**(데이터→DOM 재렌더 훅 가능)를 제공하지 않으면 빈 화면/크래시.

## 4. 검증 방법 (헤드리스 스모크 + 실 로그인)

### 4-1. 헤드리스 스모크 런처 패턴
빌드 제외(언더스코어 접두) 임시 런처를 만들어 특정 화면을 직접 띄우고 DOM/콘솔을 덤프한다. main 프로세스 복제 시 **반드시** 포함:
- `remoteMain.initialize()` + `browser-window-created`에서 `remoteMain.enable`
- `app.getAppPath` → `www` 보정
- **`global.mainRequire = require`**
- **`global.WsMsgCls = new WsMsgClsService()`** (SQLite 메시지 — 누락 시 "메시지 클래스 조회 오류" 폭주)
- 창 로드 후 `if-meta-info`로 USERINFO/SERVERINFO/THEMEINFO 주입 → DOM `executeJavaScript`로 검사

### 4-2. 검증 가능 / 불가능
| 헤드리스로 가능 | 실 로그인으로만 가능 |
|------|------|
| DOM 구조/렌더, 클래스·id | 서버 ajax 응답 기반 동작 |
| `sap is not defined` 등 동기 크래시 제거 | 미리보기 UI5 실제 렌더 |
| `node --check`, UI5 잔재 grep | 인증 **쿠키** 필요한 모든 흐름 |

> ⚠️ 가짜 USERINFO 주입만으로는 ajax가 실패해 **에러 경로만** 탄다 = 실제 동작 아님. 서버 데이터 기반 동작은 **실 로그인(인증 쿠키)으로만** 검증.

### 4-3. 실서버 디버깅
메인 창 `Ctrl+Shift+I` → Console. 미리보기 iframe 콘솔은 컨텍스트 드롭다운에서 `prevHTML` 프레임 선택. 의심 시 `getBootStrapUrl()`/`S_CODE.UA025`/`getUi5Libraries()`/`appInfo`를 임시 로그로 덤프.

## 5. WS20(디자인 영역) 변환 가이드

> WS20은 사실상 "앱 안의 앱"(원본 design 폴더 95파일/7.3만줄). **권위 스펙은 [U4A_WS20_Flow](U4A_WS20_Flow.md)/[U4A_WS20_Detail](U4A_WS20_Detail.md)**. 그 흐름(진입→LIB→getAppData→zTREE→preview→선택→attr)을 우선 기준으로 삼는다.

### 5-1. 세부 단계 (점진)
| W단계 | 내용 |
|:----:|------|
| W1 | 셸: 상단 트랜잭션 툴바 + 3분할(좌 트리/중 미리보기/우 속성) flex 레이아웃 + 빈 패널 |
| W3 | 좌측 **UI 트리**(원본 `sap.ui.table.TreeTable` → HTML5 재귀 `<ul>/<li>`), `/zTREE` 바인딩 → 수동 렌더 |
| W3.5 | 트리 **데이터 파이프라인**: `/getAppData` → T_0014 가공 → `setTreeJson` → 데코레이터 → refresh |
| W4 | 우측 **속성 패널** + **LIB 파이프라인**(`/getLibData` 6테이블 → `setCodeMasterData` → `S_CODE`) |
| W2 | 중앙 **미리보기**(iframe 호스트, KEEP-UI5) |
| W3.6+ | 트리 데코(아이콘/체크박스/행액션), 드래그&드롭, 컨텍스트 메뉴, undo/redo, 추가/삭제/이동, Binding/Event 팝업, activate/save |

### 5-2. ★ WS20 치명적 함정 (반드시 회피)
1. **대형 design 파일을 부팅 preload에 넣지 말 것**. 원래 WS20 진입 시 `getScript`로 지연 로드되는 파일(`uiDesignArea.js` 등)을 부팅 목록에 넣으면 **앱 전체가 깨진다**(모델 미생성→fnWsStart 중단). 지연 로드 또는 필요한 함수만 1:1 이식.
2. **첫 ROOT 선택은 iframe이 단독 구동**. 미리보기 iframe이 `drawPreview` 완료 후 `oLTree1.fireCellClick(ROOT)`로 첫 선택을 1회 한다. 트리/속성 측에서 **별도 ROOT 자동선택을 추가하면 충돌** → 첫 선택은 iframe의 fireCellClick → 트리 선택 핸들러 단일 경로로만.
3. **`oAPP.attr.prev` 오염 → 미리보기 자식 미표시**. `drawPreview`는 `prev`가 비어있지 않으면 "재그리기" 분기로 들어가 `APPDATA.T_0015`를 교체 → 자식 UI 임베드(`UIATY="6"`) 행 소실 → **흰 화면**(예외 없이 조용히). 미리보기 부팅 전엔 `prev`에 스탠드인을 만들지 말 것(만들면 로드 직전 purge + booting 중 차단).
4. **미리보기 동기화 호출은 iframe 준비 게이트 필수**. `refreshPreview`/`selPreviewUI`/`fn_removeMark` 등 iframe `contentWindow` 함수 호출은 `contentWindow && _loaded===true` 가드 경유. 준비 전 호출 시 iframe bare sap 크래시.
5. **공통 흐름 우회 금지**(스펙 §17): 선택=`setSelectTreeItem`, 속성변경=`attrChange→attrChangeProc→attrChgAttrVal`, 추가가능판단=공통코드(`T_0022/0023/0027/9011/S_CODE`)+`checkDenyChildAggr/checkAllowChildAggr/chkUiCardinality/designChkFixedParentUI`, Undo history, 텍스트=`fnGetMsgClsText`. 직접 `_T_0015`/row index만 건드리지 말 것.

### 5-3. 데이터 키 (실측 요약)
- `/zTREE` 노드: `OBJID/POBID/UIOBK/PUIOK/UIATK/UIATT/UILIB/ISMLB/zTREE`(+데코 `UICON/chk/highlight/visible_add/visible_delete`).
- `_T_0015` UIATY: `1`Property `2`Event `3`Aggregation `4`Association `6`Embedded.
- LIB: `T_9011`공통코드 `T_0020`라이브러리 `T_0022`Object `T_0023`Attribute `T_0024`값목록 `T_0027`상속. `T_9011`→`setCodeMasterData`→`S_CODE[CATCD]`(UA025=부트스트랩 경로 등).

## 7. 핵심 한 줄 요약
- **Electron 유지, UI5 화면만 HTML5.** 미리보기 iframe만 UI5 유지.
- **원본 무수정 + 마지막 override.** 재정의 파일은 항상 뒤에 로드.
- **시작 즉시 "기반 보정 4종"**(전역 sap 위임이 가장 중요) — 안 하면 엣지 경로에서 반드시 터진다.
- **검증 2단계**: 헤드리스(렌더·크래시) + 실 로그인(서버 동작, 인증 쿠키 필수).
- **WS20는 권위 스펙(`U4A_WS20_*`) 우선**, 첫선택 단일화·prev 오염·iframe 준비게이트 주의.

## 8. 표준 컨버전 워크플로우 (설계 산출물 기반)

> 이 절은 **AI가 단독으로 컨버전을 수행하기 위한 프로세스 메타 지침**이다. 위 [3장] 단계별 구축 순서가 "무엇을 어떤 순서로 만들지"라면, 이 절은 "각 화면을 만들 때 **어떤 설계 산출물을 작업 단위로 삼고 어떤 불변 원칙을 지킬지**"를 규정한다. 색·모양 규칙의 단일 출처는 [12_테마_컨버전_전략.md](12_테마_컨버전_전략.md), 각 기능의 변환 명세는 해당 영역 문서(01~08)의 **"AI 컨버전 실행 사양" 절**이다.

### 8.1 테마 시스템 선행/병행 구축 (12번 6장)

테마는 화면별로 즉흥 구현하지 않는다. **모든 화면 CSS가 소비할 토큰 계약을 먼저 세우거나, 첫 화면과 병행 구축**한다. 산출 순서는 [12_테마_컨버전_전략.md](12_테마_컨버전_전략.md) 6장을 단일 출처로 따른다.

기준 비주얼은 UI5 **`sap_horizon`** 이며, 테마는 **모양은 공통(horizon)·색만 다른** 5종(white/dark/purple/red/green)이다.

| 순서 | 산출물 | 역할 |
|:----:|--------|------|
| T1 | `theme/tokens.css` | 의미 토큰 계약. 색은 테마별, shape(radius/elevation/font)는 horizon 공통 고정. 모든 컴포넌트 CSS가 이 키만 소비 |
| T2 | `theme/shell.css` | 셸 공통 컴포넌트 CSS. `var(--*)`만 소비, hex 색상·하드코딩 radius·shadow **0건** |
| T3 | `theme/themes/{white,dark,purple,red,green}.css` | `[data-theme="..."]` 색 토큰 세트. 중립색은 `sap_horizon`/`sap_horizon_dark` 덤프에서 이식, 강조색만 테마별(추측 금지). `dark`만 배경·텍스트·경계까지, `purple/red/green`은 `--accent` 램프만 override |
| T4 | `theme/theme-api.js` | `U4ATheme.apply / param / onChange` shim. UI5 `applyTheme`/`Parameters.get`/`attachThemeChanged` 호출부를 동일 시그니처로 대체. 기본 테마 white |

> 토큰 계약(T1)이 없으면 화면 CSS가 색을 하드코딩하게 되어 이후 드리프트 정리 비용이 커진다. **따라서 토큰 계약은 첫 화면(02 로그인/서버리스트)보다 앞서거나 늦어도 동시에 확정**한다.

### 8.2 각 기능 문서의 "AI 컨버전 실행 사양" 절 = 단일 작업 단위

각 영역 문서(01~08)에는 표준 **"AI 컨버전 실행 사양"** 절이 있고, 이 절 하나가 해당 화면 변환의 **완결 가능한 작업 단위**다. 화면 하나를 변환할 때는 이 절을 통째로 입력으로 삼아 다음 5요소를 순서대로 소화한다.

| 구성요소 | 내용 | AI 활용 |
|---|---|---|
| **시안·참조 출처** | 분석 장(기능/모델) + UI 매핑 장 + `.codex_tmp` 1차 시안 + 테마 사양 | 어디서 무엇을 가져올지 |
| **산출물 파일 트리** | target 경로별 산출 파일 | 무엇을 만들지 |
| **작업 순서(입력→산출→수용 기준)** | 단계별 체크포인트 | 어떻게 진행·검증할지 |
| **불변 제약** | 보존해야 할 Electron/Node·인증·메타 계약 | 무엇을 건드리면 안 되는지 |
| **AI 실행 프롬프트** | 새 세션에 그대로 투입 가능한 지시문 | 작업 착수 트리거 |

> 화면 변환 요청을 받으면 **그 화면 문서의 "AI 컨버전 실행 사양" 절 + 12번 테마 문서**를 함께 읽고, 두 문서를 모두 적용해야 비로소 완성이다(예: 02번은 9장, 테마는 12번 6장).

### 8.3 공통 불변 원칙 체크리스트 (모든 화면 공통)

화면을 막론하고 아래는 항상 성립해야 한다. 각 단계 산출 후 self-check 한다. (영역별 세부 불변 제약은 해당 문서 "AI 컨버전 실행 사양"의 불변 제약 항을 추가로 따른다.)

- [ ] **기능 동등성 — 기존 기능 전수 보존** — 기존 기능·이벤트·단축키·검증·상태·서버분기·엣지케이스를 **하나도 빠짐없이 동등하게** 재현한다(외형이 아니라 행동 기준). 각 기능 문서의 "기능 동등성 — 기능 인벤토리" 절로 전수 추출·매핑·검증(누락 0). 임의 단순화·동작 생략 금지.
- [ ] **Electron/Node 자원 유지** — `BrowserWindow`/`@electron/remote`/IPC/`fs`/`net`(Named Pipe)/`regedit`/Web Worker/SAP 인증 XHR 등 호출부를 그대로 둔다. **UI 레이어만** 교체([0장], 02번 8장 (A)).
- [ ] **색·모양은 토큰만, 하드코딩 금지** — 모든 색·radius·shadow는 `theme/tokens.css`의 의미 토큰(`var(--*)`)을 소비한다. hex 하드코딩·동일 의도 색의 드리프트 금지([12_테마_컨버전_전략.md](12_테마_컨버전_전략.md) 5장·6.4).
- [ ] **흔히 빠지는 표면 전수 점검** — 스크롤바·드롭다운 펼침·focus-visible 링·hover/disabled/selected·ValueState·placeholder·`::selection`·모달 backdrop·오버레이·비활성 창 등. 정지 화면만 보지 말고 hover/focus/disabled/error/open 상태를 직접 확인한다([12_테마_컨버전_전략.md](12_테마_컨버전_전략.md) 6.6 체크리스트 A~I).
- [ ] **테마 전환은 shim 경유** — `applyTheme(...)` 대신 `U4ATheme.apply(name)`, 메타로 전달되는 `THEMEINFO.THEME`을 `data-theme`로 매핑(`sap_horizon→white`/`sap_horizon_dark→dark` 등, 기본 white)([12_테마_컨버전_전략.md](12_테마_컨버전_전략.md) 5.2).
- [ ] **Preview iframe은 UI5 테마 유지** — 미리보기(`design/preview/`)는 실제 UI5가 도는 영역이므로 셸 테마 시스템을 침투시키지 않는다([1장] 전략 B, 12번 1.1·6.4).
- [ ] **텍스트는 메시지키로** — 화면 문구는 메시지 키(`fnGetMsgClsText` / `WsMsgCls`)로 처리하고 하드코딩하지 않는다. 테마는 색/모양만 담당(12번 6.4).
- [ ] **메타·인증 계약 불변** — `oMetadata`(`SERVERINFO`/`THEMEINFO`/`EXEPAGE`/…) 구조와 `if-meta-info` IPC 채널, 인증·세션 흐름(XHR `/wsloginchk`, `withCredentials`, `session.partition`, 세션 워커)은 바꾸지 않는다(02번 9.4).
