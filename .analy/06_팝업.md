> 대상 버전: U4A Workspace v3.6.3  
> 분석 기준: `www/ws30/ws10_20/Popups/`, `www/ws30/ws10_20/js/fn*PopupOpen*.js`, `fnDialogPopupOpener.js`

## 1. 개요 — 팝업 아키텍처

### 두 가지 방식

| 구분 | 방식 | 특징 |
|------|------|------|
| **BrowserWindow 팝업** | Electron `new REMOTE.BrowserWindow(...)` | 별도 OS 프로세스(창)로 뜸. 독립적인 HTML 파일 로드. IPC로 메인 창과 통신 |
| **UI5 Dialog 팝업** | `new sap.m.Dialog(...)` / `sap.m.MessageBox` 등 | 메인 창의 DOM 위에 오버레이. 같은 JS 컨텍스트 공유. 콜백으로 데이터 반환 |

### BrowserWindow 팝업 공통 생성 패턴

모든 BrowserWindow 팝업은 아래 흐름을 따른다:

```
1. fnSetBusyLock("X")                  ← 메인 창 잠금
2. oMainBroad.postMessage({BUSY_ON})   ← 하위 창들에 브로드캐스트
3. getCheckAlreadyOpenWindow(이름)     ← 이미 열렸으면 포커스만
4. new REMOTE.BrowserWindow(opts)      ← 창 생성 (opacity=0, show=false)
5. oBrowserWindow.loadURL(sLoadUrl)    ← QueryString 포함 URL 로드
6. 'did-finish-load' 이벤트          ← IPC로 초기 데이터 전송
7. setParentCenterBounds(...)          ← 부모 창 기준 중앙 배치
8. setBrowserOpacity(oBrowserWindow)   ← opacity 0→1 페이드인 연출
9. 'closed' 이벤트                   ← IPCMAIN 이벤트 해제, 부모 포커스 복귀
```

### opener → frame 구조

대부분의 BrowserWindow 팝업은 2단 iframe 구조를 사용한다:

```
[메인 창 (WS10/WS20/WS30 페이지)]
  └─ fnXxxPopupOpener() 호출
       └─ BrowserWindow 생성
            └─ index.html (진입 페이지)
                 └─ <iframe id="mainFRAME" src="frame.html">
                       └─ frame.js (실제 팝업 로직, UI5 or 바닐라 JS)
```

단독 구조 팝업(frame 없음): `iconPrevPopup`, `errMsgPopup`, `winShowHidePopup`, `runtimeClassNavigator` 등은 index.html/index.js에서 직접 UI5 렌더링.

### Mutation 감지와 예외처리

`sap.m.BusyDialog`, `sap.m.MessageBox`, `sap.m.Dialog` 등 UI5 Dialog 계열 UI가 DOM에 추가되면 Mutation Observer가 감지하여 현재 떠있는 자식 BrowserWindow들을 자동으로 숨긴다(hide). 해당 Dialog 닫힘 시 다시 활성화(show)된다.  
예외 처리가 필요한 경우: `oDialog.data("MUTATION_EXCEP", "X")` 커스텀 데이터 설정.

## 2. 팝업 생성 공통 메커니즘

### 2-1. fnDialogPopupOpener.js — 중앙 디스패처

`www/ws30/ws10_20/js/fnDialogPopupOpener.js` (3623줄)에 모든 팝업 opener 함수가 정의된다.  
각 함수는 지연 로딩(lazy-load) 패턴으로 구현된다:

```javascript
oAPP.fn.fnXxxPopupOpener = function (...args) {
    if (oAPP.fn.fnXxxPopupOpen) {        // 이미 로드됐으면
        oAPP.fn.fnXxxPopupOpen(...args); // 바로 실행
        return;
    }
    oAPP.loadJs("fnXxxPopupOpen", function () { // 지연 로드
        oAPP.fn.fnXxxPopupOpen(...args);
    });
};
```

UI5 Dialog 방식 팝업은 `fnDialogPopupOpener.js` 내에서 직접 `new sap.m.Dialog(...)` 생성 없이 별도 `fnXxxPopupOpen.js` 파일로 위임한다.

### 2-2. popup_modules/_module_dialog_pattern

`www/ws30/ws10_20/js/popup_modules/_module_dialog_pattern/`  
BrowserWindow 팝업을 만드는 표준 템플릿 코드(index.js)가 포함된 샘플 모듈.  
실제 팝업 index.js들은 이 패턴을 복사·커스터마이징하여 구현한다.

### 2-3. Electron Popup 고려사항 (www/_doc/Electron Popup 만들때 고려 사항.txt)

> 부모 지정 없이 띄우는 팝업의 경우 `oAPP.fn.closeAllCurrWinDependentPopups()` 로직 위치를 반드시 확인해야 한다.

- `parent` 없이 생성된 팝업은 메인 창이 닫혀도 독립적으로 남을 수 있음.
- 메인 창 `pagehide` 이벤트에서 팝업을 닫는 코드(`window.addEventListener('pagehide', ...)`)를 반드시 추가.

## 4. 주요 팝업 상세

### 4-1. editorPopup — 소스 에디터

**진입**: `fnEditorPopupOpener(oEditorInfo, sSearchValue)` → `fnEditorPopupOpen.js`

- `oEditorInfo.OBJTY`: 에디터 구분 키 (예: `CS`=Custom Style, `HM`=HTML, `JS`=JavaScript 등)
- 이미 열린 창이면 포커스만 주고, `OBJTY === "CS"`인 경우 검색값을 해당 창에 IPC로 전달
- 창 구조: `editorFrame.html` → `editor.html` (Ace Editor 탑재)
- IPC 채널: `did-finish-load` → `if-editor-info` 전송으로 초기 소스코드 로드
- 저장 시: IPC `${BROWSKEY}--editor-save` 로 메인 창에 반환

**editorPopup_v2**: Monaco Editor 기반 실험 버전. `module.exports = function(REMOTE, oAPP, oParams)` 형태로 index.js에서 직접 창 생성. 현재 개발자 PC 환경에서만 분기 활성화(주석 처리).

---

### 4-2. bindPopup — 데이터 바인딩 편집

**진입**: `fnBindWindowPopupOpener()` (fnDialogPopupOpener.js 내 직접 구현)

- SAP 데이터 모델(T_9011, T_0022, T_0023, T_0014, T_0015, T_CEVT)을 IPC `if_modelBindingPopup`으로 전달
- WS20 디자인 화면과 BroadcastChannel(`broadcastChannelBindPopup.js`)로 양방향 통신
- `parent = CURRWIN` 지정, `closable = false`로 모달처럼 동작
- 닫힐 때 `CHANNEL-CLOSE` 호출로 채널 종료, IPC `if-bindPopup-callback` 이벤트 해제

---

### 4-3. findPopup — UI 컨트롤 찾기

**진입**: `fnFindPopupOpener()` → `fnFindPopupOpen.js`

- `sPopupName = "UIFIND"`, BrowserWindow 방식
- `fnFindPopupOpen.js` 내부에서도 UI5 Dialog(`sap.m.Dialog`)를 사용하여 검색 조건 입력 UI 구성 (UI5 Dialog 방식 채택 - WS20 컨텍스트에서 실행)
- 검색 결과를 `IPCMAIN.on(...)` 이벤트로 메인 창에 전달

**참고**: 이 팝업은 `fnFindPopupOpen.js`가 UI5 Dialog를 사용하므로, 실질적으로 **UI5 Dialog 방식**이다. BrowserWindow는 WS30 context에서 호출될 때만 사용.

---

### 4-4. importExportPopup — 앱 Import/Export

**진입**: `fnWsImportExportPopupOpener(sFlag)` — `sFlag`: `"IMPORT"` 또는 `"EXPORT"`

- EXPORT 시: 앱 이름 유효성 검사 → 서버에서 앱 존재 여부 확인 → 팝업 열기
- `transparent: true, frame: false` 설정으로 커스텀 프레임 UI
- IPC: `export_import-INITDATA` (초기데이터), `${BROWSKEY}--export_import-IMPORT`, `${BROWSKEY}--export_import-EXPORT`

---

### 4-5. optionPopup — Workspace 옵션

**진입**: `fnWsOptionsPopupOpener()`

- 창 이름: `"WSOPTS"`, `optionM.html` / `optionS.html` 2개 파일
- 전달 데이터: `BROWSKEY`, `oUserInfo`, `oServerInfo`, `SYSID`, `THEME_INFO`, `ISCDN`
- IPC 채널: `if-ws-options-info`, `${BROWSKEY}-cdn-save`

---

## 6. HTML5 컨버전 시 고려사항

> **전제**: 이 앱은 계속 **Electron 데스크톱 앱**으로 유지된다.  
> 컨버전 대상은 **팝업 내부 UI 렌더링이 SAP UI5로 구현된 부분**뿐이다.  
> BrowserWindow 팝업 생성 방식, IPC 데이터 전달, BroadcastChannel, `parent.require`, `setContentProtection` 등 모든 Electron/Node 자원은 **그대로 재사용**한다.

---

### 6-1. (A) 그대로 유지할 항목 — Electron/Node 자원

| 항목 | 현재 구현 | 유지 이유 |
|------|----------|----------|
| **BrowserWindow 팝업 생성** | `new REMOTE.BrowserWindow(opts)` | 앱이 Electron으로 유지되므로 OS 창 방식 그대로 사용 |
| **팝업 URL 로드** | `oBrowserWindow.loadURL(sLoadUrl)` | 파일 경로 기반 로드, 변경 불필요 |
| **부모 창 지정** | `parent = CURRWIN`, `modal = true` | OS 레벨 모달 동작 보장, 유지 |
| **IPC 초기 데이터 전송** | `webContents.send('CHANNEL', data)` / `IPCRENDERER.on(...)` | 메인↔렌더러 프로세스 간 통신, 유지 |
| **IPCMAIN 이벤트 등록/해제** | `IPCMAIN.on(...)` / `IPCMAIN.off(...)` | 메인 프로세스에서 처리, 유지 |
| **BroadcastChannel** | `BroadcastChannel`로 WS20 ↔ bindPopup 양방향 통신 | Web BroadcastChannel API는 Electron에서도 그대로 동작 |
| **parent.require** | `parent.require(sPath)` — 팝업에서 메인 창 Node 모듈 접근 | nodeIntegration 환경 유지 시 그대로 사용 가능 |
| **setContentProtection** | `win.setContentProtection(true)` — screen_record 인디케이터 창 | OS 레벨 캡처 차단, Electron 전용 기능으로 유지 |
| **투명·프레임리스 창** | `transparent: true, frame: false` — importExportPopup, textSearchPopup 등 | Electron BrowserWindow 옵션, 유지 |
| **중복 창 방지** | `getCheckAlreadyOpenWindow(name)` | 현재 구조 그대로 사용 |
| **부모 창 중앙 배치** | `WSUTIL.setParentCenterBounds(REMOTE, win)` | REMOTE.screen API 사용, Electron 유지 |
| **opacity 페이드인** | `WSUTIL.setBrowserOpacity(win)` | BrowserWindow.setOpacity(), 유지 |
| **창 플래시** | `win.flashFrame(true)` | OS 작업표시줄 깜빡임, Electron 전용 |
| **Mutation Observer + 자식 창 hide/show** | UI5 Dialog DOM 감지 → 자식 BrowserWindow hide | 현재 구조 그대로 유지 |
| **pagehide 창 닫기** | `window.addEventListener('pagehide', ...)` | 메인 창 종료 시 팝업 정리 로직, 유지 |
| **QueryString 파라미터 전달** | `browserkey`, `sessionKey`, `OBJTY`, `USERINFO` 등 URL 파라미터 | 팝업 HTML 로드 시 그대로 사용 |

### 6-2. (B) HTML5로 교체할 UI5 요소 — 내부 렌더링

아래는 팝업 내부 렌더링이 UI5로 구현된 경우, 해당 UI 레이어를 순수 HTML5+CSS+바닐라 JS로 교체할 때의 분류이다. **BrowserWindow 자체는 유지**하고, 팝업 창 내부의 sap-ui-bootstrap 로드 및 sap.m.* 컨트롤만 교체한다.

#### UI5 의존도 · 교체 난이도 분류

| 난이도 | 팝업 | UI5 사용 범위 | HTML5 교체 방향 |

## 7. AI 컨버전 실행 사양 (팝업)

> 이 절은 **설계 산출물**이다. 실제 구현은 추후 진행하며, 그 시점에 AI 는 이 절을 읽고 **기능 컨버전 + 테마 적용**까지 스스로 완결한다.  
> 기능 매핑은 6장(6-1 유지 / 6-2 교체)을, 테마는 [12_테마_컨버전_전략.md](12_테마_컨버전_전략.md) 를 따른다. 두 문서를 함께 적용해야 완성이다.  
> 팝업은 **Electron 별창 팝업(`BrowserWindow`)** 과 **인앱 `sap.m.Dialog` 계열** 두 방식이 섞여 있으므로(1장), 대상 팝업이 어느 방식인지 먼저 식별하고 그에 맞는 산출물·작업 순서를 적용한다.

### 7.2 산출물 파일 트리 (target)

> 팝업은 폴더 단위로 산재한다(3장 카탈로그·5장 opener 표 기준). 아래는 **대표 팝업 한 종을 컨버전할 때의 산출 형태**이며, 각 팝업 폴더에 동일 패턴을 적용한다.

```
www/ws30/ws10_20/Popups/<팝업폴더>/
├─ index.html / frame.html   ← 6-2 매핑대로 마크업 (sap-ui-bootstrap 제거, UI5 컨트롤 → HTML5)
├─ index.js   / frame.js     ← JSONModel 바인딩 → JS 상태객체 + render 함수. BrowserWindow 생성·IPC·콜백은 유지(6-1)
└─ css/
   └─ <팝업>.css             ← 컴포넌트 CSS, theme/tokens.css 의 var(--*)만 소비 (색·radius·shadow 하드코딩 0)

# 방식별 분기
# (a) Electron 별창 팝업(BrowserWindow): 위 폴더 그대로. 창 생성/loadURL/IPC/BroadcastChannel 유지
# (b) 인앱 sap.m.Dialog 계열(fnXxxPopupOpen.js): 별도 창 없음 → 부모 DOM 내 HTML5 <dialog> 요소로 교체

www/ws30/ws10_20/theme/      ← 12번 문서 6.1 산출물 (선행 또는 병행)
├─ tokens.css  ├─ shell.css  ├─ theme-api.js  └─ themes/*.css
```

> 팝업 HTML/CSS 는 별창이든 인앱 `<dialog>` 든 **`theme/tokens.css` 의 토큰을 소비**한다. 테마 시스템(12번 6장)이 아직 없으면 토큰 계약(12번 6.2)을 먼저 세운 뒤 진행한다. 별창 팝업은 자체 문서이므로 `theme/tokens.css`·`themes/*.css`·`theme-api.js`를 팝업 HTML 에서도 로드하고, 부모로부터 전달받은 테마명을 적용해 **부모와 동일 테마 일관성**을 보장한다.

### 7.2.1 기능 동등성 — 기능 인벤토리 (전수 보존)

### 7.3 작업 순서 (각 단계 = 입력 → 산출 → 수용 기준)

1. **팝업 방식 식별 + Electron/Node 자원 보존** — 대상 팝업이 (a) 별창(BrowserWindow) 인지 (b) 인앱 `sap.m.Dialog` 인지 3·5장으로 판별. 6-1 (A) 의 자원(`new REMOTE.BrowserWindow`, `loadURL`, IPC `webContents.send`/`IPCMAIN.on/off`, `BroadcastChannel`, `parent.require`, `setContentProtection`, `getCheckAlreadyOpenWindow`, opacity/center 유틸, `pagehide` 정리)은 **호출부 그대로 유지**. 수용: 6-1 (A) 항목 코드 변경 0건, 별창 팝업은 BrowserWindow 생성 흐름(1장 9단계) 보존.
2. **UI 마크업 교체** — 6-2 매핑대로 sap-ui-bootstrap 제거, UI5 컨트롤 → HTML 요소. 특히 팝업 내부 서브 모달(`sap.m.Dialog`/`sap.m.MessageBox`) → HTML5 `<dialog>` (`showModal()`/`close()`). 수용: 팝업 화면에서 UI5 컨트롤(`sap.m.*`) 미생성, `<dialog>` 로 동작.
3. **상태/렌더 분리** — JSONModel 바인딩 → JS 상태객체 + `render*()` 함수(6-2). 수용: 모델 변경 시 DOM 동기화 동작, 대형 목록 forEach 렌더 정상.
4. **테마 토큰 소비** — 모든 색·radius·shadow 를 `theme/tokens.css` 의 의미 토큰(12번 6.2)으로. 수용: 팝업 CSS 내 hex 색상·하드코딩 radius·shadow **0건**(grep 검증) **+ 12번 6.6 체크리스트 표면**(스크롤바·드롭다운 펼침·focus 링·hover/disabled·ValueState·placeholder·모달 backdrop/오버레이 등) 전수 토큰화.
5. **테마 전환 연동 + 팝업 테마 일관성** — `applyTheme` 대신 `U4ATheme.apply(name)`(12번 5.2). 팝업이 IPC/QueryString 으로 전달받는 `THEME`/`THEME_INFO`(예: optionPopup·patternPopup·uspNewPopup·illustMsgPopup 의 테마 전달)를 `data-theme` 로 매핑. 별창 팝업도 부모와 동일 토큰 세트 로드. 수용: 부모 테마 변경 시 별창/인앱 팝업 색상이 **동일 테마로 함께** 변경.
6. **텍스트 비하드코딩** — 화면 문구는 메시지 키(`oAPP.common.fnGetMsgClsText` 등 또는 JS 메시지 객체/JSON fetch)로. 수용: 팝업 문구 하드코딩 없음.

### 7.4 불변 제약 (반드시 보존)

- **Electron 별창 팝업 생성·IPC 계약 유지** — 6-1 (A). 특히 `new REMOTE.BrowserWindow` 옵션(`parent`/`modal`/`transparent`/`frame`), `loadURL` QueryString, `did-finish-load` 초기 데이터 IPC, `IPCMAIN.on/off` 채널명, `getCheckAlreadyOpenWindow` 중복 방지, `pagehide` 정리. 채널명·이벤트명을 바꾸지 않는다.
- **팝업↔부모 데이터 콜백 계약 유지** — `webContents.send`/`IPCRENDERER.on` 콜백, `BroadcastChannel`(bindPopup ↔ WS20 양방향), ui5CssPopup_v2 의 함수 콜백(`fnCallback(oRes)`) 등 반환 계약. UI만 교체하고 데이터 흐름은 건드리지 않는다.
- **테마는 12번 문서 단일 출처 + 팝업 테마 일관성** — 색/모양은 토큰만, 하드코딩·드리프트 금지. 별창 팝업도 부모와 같은 테마명을 적용해 일관성을 유지한다. Preview iframe(WS20)은 UI5 테마 유지(이 팝업들엔 해당 없음).
- **Electron/Node 자원 불변** — `setContentProtection`(screen_record), `flashFrame`, Mutation Observer 기반 자식 창 hide/show, `parent.require`, opacity 페이드인·중앙 배치 유틸 등 Electron 전용 기능을 제거하지 않는다.
- **기능 동등성** — 기존 기능·이벤트·단축키·검증·상태·콜백 계약·엣지케이스를 빠짐없이 동등 재현(행동 기준). 임의 단순화·생략 금지.
