> **분석 대상 버전**: U4A Workspace v3.6.3  
> **분석 기준 브랜치**: analysis  
> **작성일**: 2026-06-09

---

## 1. 개요

로그인 & 서버 리스트 영역은 U4A Workspace의 진입점이다. 사용자가 SAP 시스템에 접속하기 위해 거치는 두 단계 화면으로 구성된다.

1. **서버 리스트 화면 (ServerList)**: Electron 앱 최초 실행 시 노출. SAP Logon Pad의 `SAPUILandscape.xml`을 읽어 SAP 서버 목록을 표시하고, 사용자가 접속할 서버를 선택하면 새 BrowserWindow를 생성하여 로그인 화면으로 이동한다.
2. **로그인 화면 (Login)**: 선택한 SAP 시스템에 대해 Client / ID / Password / Language를 입력받아 SAP 서버에 POST 인증을 수행한다. 성공 시 WS10(앱 검색) 화면으로 이동한다.

### 아키텍처 개요

```
[Electron Main Process]
    └── BrowserWindow (ServerList.html)  ← SAPUILandscape.xml 파싱, 서버 목록 표시
            └── 서버 선택 → new BrowserWindow (Login.html) 생성
                    └── XHR POST /wsloginchk → SAP 서버 인증
                            └── 성공 → loadWS30MainPage() (WS10 화면)
```

### 주요 Electron/Node 의존 항목 요약

| 분류 | 현재 사용 기술 | 유지 여부 |
|------|---------------|-----------|
| 창 관리 | `@electron/remote`, `REMOTE.BrowserWindow` | **유지** (Electron 데스크톱 앱) |
| 파일 시스템 | `FS.readFileSync`, `FS.writeFileSync` | **유지** (p13n, 설정 파일 I/O) |
| 레지스트리 | `regedit`, Windows Registry | **유지** (SAP Logon 경로 자동 취득) |
| IPC 통신 | `ipcMain`, `ipcRenderer` | **유지** (창 간 메시지 전달) |
| Named Pipe | Node.js `net` 모듈, Windows Named Pipe | **유지** (U4A EDU 외부 프로세스 연동) |
| XML 파싱 | `xml-js` npm 패키지 | **유지** (SAPUILandscape.xml 파싱) |
| 자동 업데이트 | `electron-updater`, `autoUpdaterSAP` | **유지** (Electron 앱 업데이트) |
| 글로벌 단축키 | `GlobalShortCut`, `REGEDIT` | **유지** (데스크톱 전역 단축키) |
| SAP 인증 XHR | `xhr.withCredentials`, 세션 쿠키 | **유지** (SAP 서버 POST 인증) |
| Web Worker | `new Worker(...)` | **유지** (세션 유지 워커) |

### 2.2 서버 등록/관리

서버 목록은 **SAPGUI의 `SAPUILandscape.xml` 파일을 읽어** 표시한다. 직접 서버를 등록하는 것이 아니라 SAPGUI Logon에 이미 등록된 서버를 가져온다.

#### 데이터 소스 경로 취득 흐름

```
oAPP.fn.fnOnListupSapLogon()
  └── oAPP.fn.fnGetRegInfoForSAPLogon()
        └── REGEDIT.list(SETTINGS.regPaths.saplogon, ...)
              └── 레지스트리 경로: HKCU\SOFTWARE\SAP\SAPLogon (또는 유사 경로)
                    └── result.LandscapeFile.value  → SAPUILandscape.xml 절대경로
  └── oAPP.fn.fnGetRegInfoForSAPLogonThen()
        └── FS.watch(sLandscapeFilePath, fnSapLogonFileChange)  // 파일 변경 감지
        └── oAPP.fn.fnReadSAPLogonData("LandscapeFile", sLandscapeFilePath)
              └── FS.readFileSync() → XMLJS.xml2js() → JSON 파싱
        └── oAPP.fn.fnReadSAPLogonDataThen()
              └── fnCheckSapguiVersion()  // SAPGUI 버전 체크 (PowerShell 스크립트 실행)
              └── fnSetSAPLogonLandscapeList()  // 모델에 서버 목록 바인딩
```

#### 서버 목록 JSON 구조 (파싱 후 `/ServerList` 모델 경로에 저장)

```json
[
  {
    "uuid":     "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
    "name":     "서버 표시명",
    "systemid": "NEP",
    "host":     "192.168.1.100",
    "port":     "3200",
    "insno":    "00",
    "mode":     "0",
    "server":   "logon_group명 (메시지 서버 방식)",
    "routerid": "라우터 UUID (있을 경우)",
    "msid":     "메시지 서버 UUID (있을 경우)",
    "router":   { "host": "...", "port": "..." },
    "msgsvr":   { "host": "...", "port": "3600" }
  }
]
```

`insno`(인스턴스 번호)는 `port.substring(2, 4)`로 추출한다. 예: port `3200` → insno `00`.

#### 추가 저장 데이터 (로컬 JSON 파일: `SERVERINFO_V2`)

SAPGUI XML에는 없는 WS 전용 설정(protocol, host override, port override 등)을 별도 JSON 파일에 UUID 키로 저장한다.

```json
[
  {
    "uuid":     "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
    "protocol": "https",
    "host":     "sapdomain.example.com",
    "port":     "443",
    "settings": {
      "useInternal": false
    }
  }
]
```

### 2.3 SAP Landscape 연동

- SAPGUI 설치 여부는 **Windows Registry**(`HKCU\SOFTWARE\SAP\SAPLogon`)를 통해 확인한다.
- `SAPUILandscape.xml`을 `xml-js` 패키지로 파싱한다.
- XML 내 `<Services>` → `<Service>` 항목이 실제 서버 목록이다.
- `<Routers>`, `<Messageservers>` 항목을 별도로 추출하여 서비스 항목에 join한다.
- 메시지 서버 포트가 없을 경우 `C:\Windows\System32\drivers\etc\services` 파일에서 `sapms<SYSID>` 항목을 `child_process.exec`으로 `findstr` 명령 실행하여 추출한다.
- SAPGUI 버전 체크는 PowerShell 스크립트(`get_sapgui_inf.ps1`)를 `child_process.spawn`으로 실행한다. 버전 7700 미만이면 앱을 강제 종료한다.
- `FS.watch()`로 `SAPUILandscape.xml` 변경을 감지하며, 변경 시 1초 디바운스 후 목록을 자동 갱신한다.

### 2.5 네트워크 체크

`fnNetworkChecker.js`가 담당한다. 로직은 단순하며 브라우저 표준 이벤트를 사용한다.

```javascript
// 네트워크 복구 시
oAPP.fn.fnNetworkCheckerOnline = function() {
    oAPP.attr.bIsNwActive = true;
    oAPP.common.fnIsChildWindowShow(true);   // Child Window 활성화
    parent.setNetworkBusy(false);            // 네트워크 바쁨 표시 해제
};

// 네트워크 단절 시
oAPP.fn.fnNetworkCheckerOffline = function() {
    oAPP.attr.bIsNwActive = false;
    oAPP.common.fnIsChildWindowShow(false);  // Child Window 비활성화
    parent.setNetworkBusy(true, iZindex);    // 네트워크 바쁨 표시
};
```

이 함수는 WS10/20 공통 main 프레임에서 `window.addEventListener('online', ...)` / `window.addEventListener('offline', ...)`에 연결된다.

### 3.2 로그인 UI 구성 (SAP UI5 기반)

`oAPP.fn.fnGetLoginForm()`이 `sap.ui.layout.form.Form`을 생성하며 다음 필드를 포함한다.

| 필드 ID | 컨트롤 | 바인딩 경로 | 비고 |
|---------|--------|------------|------|
| `ws_client` | `sap.m.Input` (Number) | `/LOGIN/CLIENT` | SAP 클라이언트 (3자리) |
| `ws_id` | `sap.m.SearchField` | `/LOGIN/ID` | SAP 사용자 ID, 자동완성 기능 포함 |
| `ws_pw` | `sap.m.Input` (Password) | `/LOGIN/PW` | 비밀번호, 보기/숨기기 토글 가능, CapsLock 감지 |
| `ws_langu` | `sap.m.Input` | `/LOGIN/LANGU` | SAP 접속 언어 (직접 입력, 기본 hidden) |
| `ws_langu_select` | `sap.m.Select` | `/LOGIN/LANGU` | SAP 접속 언어 (셀렉트 방식, 기본 hidden) |
| `ws_wslangu` | `sap.m.Select` | `/LOGIN/WSLANGU` | Workspace UI 표시 언어 (EN/KO) |

| `ws_loginBtn` | `sap.m.Button` | - | 로그인 버튼, `ev_login` 이벤트 연결 |

로그인 Form은 `sap.f.Card`로 감싸서 헤더에 U4A 로고와 타이틀을 표시한다.

커스텀 헤더에는 창 최소화/최대화/닫기 버튼이 포함되며, `CURRWIN.minimize()`, `CURRWIN.maximize()`, `CURRWIN.close()` (Electron API)를 직접 호출한다.

### 3.4 언어 설정

두 가지 언어를 별도로 관리한다.

| 언어 종류 | 필드 | 설명 |
|----------|------|------|
| **SAP 접속 언어** (`LANGU`) | `ws_langu` / `ws_langu_select` | SAP 서버 세션 언어. 서버에 `sap-language`로 전달됨 |
| **Workspace UI 언어** (`WSLANGU`) | `ws_wslangu` | WS 인터페이스 표시 언어 (EN/KO). 로그인 성공 후 UI5 부트스트랩 언어에 적용 |

#### 언어 설정 관련 이력 (`글로벌 언어 설정 관련.txt` 기준)

- 과거에는 `parent.process.isServDependLangu` 전역변수로 "서버 접속 언어 기준 표시" 여부를 제어했다.
- 현재(v3.6.x)는 이 전역변수를 제거하고 두 언어를 명확히 분리하였다.
- `fnLoadBootStrapSetting()`에서 `WSUTIL.getWsSettingsInfo().globalLanguage`를 읽어 UI5 `data-sap-ui-language`에 적용한다.
- 로그인 성공 후 `oUserInfo.LANGU = oUserInfo.WSLANGU`로 덮어써서, WS 내부에서의 언어는 WSLANGU 기준으로 동작한다.

### 3.6 Trial(체험판) 버전 분기

`Login_trial.js`가 trial 전용 UI를 담당한다.

- `oWsSettings.isTrial === true`일 경우 `fnOnTrialLoginPageRendering()`으로 Guest Login 화면을 렌더링한다.
- Guest Login은 SAP 아이콘 Avatar 클릭 시 서버 연결 가능 여부를 먼저 확인(`fnCheckServerAvaliable`)한다.
- 서버 연결 확인 후 `trialServerInfo`(settings.json에 하드코딩)에서 CLIENT, ID, PW, LANGU, SYSID를 자동 입력하여 `ev_login()`을 호출한다.
- Trial 버전에서는 `oResult.META.HOST`를 `trialServerInfo`의 SERVERIP/INSTANCENO로 강제 오버라이드한다.

### 3.7 쿠키/세션 처리

- `xhr.withCredentials = true`로 SAP 세션 쿠키를 자동 포함한다.
- 로그인 성공 시 `oResult.HTTP_ONLY` 값을 `oAPP.attr.HTTPONLY`에 저장한다. 이 값이 `"1"`이면 HttpOnly 쿠키로 세션이 관리되며, 이후 세션 유지를 위해 사용자 ID/PW를 추가로 전송해야 하는 로직이 존재했으나, 현재는 주석 처리된 상태이다.
- 각 BrowserWindow는 고유 `partition`(= `SESSKEY`)을 가지며, Electron의 세션 격리 기능을 활용한다.
- `configureSession(oBrowserWindow)` 함수가 SameSite 쿠키 관련 Electron 세션 설정을 조정한다.

### 3.8 Remember(자동 입력) 처리

로그인 정보는 `USERDATA/p13n/login.json` 파일에 SYSID 별로 저장된다.

```json
{
  "SY1": {
```

## 5. 세션 유지 메커니즘

### 5.1 fnServerSession.js

`www/ws30/ws10_20/js/fnServerSession.js`가 세션 유지를 담당한다.

```javascript
oAPP.fn.fnServerSession = function(bIsForceRun) {
    var oSendParam = {
        SERVPATH: parent.getServerPath() + "/dummycall",
        USERINFO: parent.getUserInfo()
    };

    // Web Worker 생성
    oAPP.attr._oServerWorker = new Worker("./js/workers/u4aWsServerSessionWorker.js");
    oAPP.attr._oServerWorker.onmessage = oAPP.fn.fnServerSessionTimeOut;
    oAPP.attr._oServerWorker.postMessage(oSendParam);
};
```

### 5.2 u4aWsServerSessionWorker.js

Web Worker로 동작하며, 10분 간격으로 `/dummycall` 엔드포인트에 `POST XHR`을 보내 SAP 세션을 유지한다.

```
세션 유지 워커 동작 흐름:
  onmessage({ SERVPATH, USERINFO })
    └── setInterval(10분)
          └── XHR POST /dummycall
                ├── 성공 (TYPE != "E") → 계속 유지
                └── 실패 (TYPE == "E" or HTTP 오류)
                      └── self.postMessage({ RETCD: "E", RTMSG: "..." })
                            └── fnServerSessionTimeOut() 실행
                                  ├── IPCRENDERER.send('if-browser-close', ...)  // 다른 창 종료
                                  └── 세션 만료 Dialog 표시 (fnShowIllustMsgDialog)
                                        └── 확인 시 fn_logoff_success() → 로그아웃
```

세션 만료 시 `RETCD: "E"` 메시지를 `postMessage`로 메인 스레드에 전달하고, 메인 스레드에서 다른 모든 BrowserWindow를 IPC로 종료시킨 후 로그인 화면으로 이동한다.

### 9.3 작업 순서 (각 단계 = 입력 → 산출 → 수용 기준)

1. **Electron/Node 자원 보존 확인** — 8장 (A) 의 자원(`REMOTE.BrowserWindow`, `FS`, `regedit`, Named Pipe, XHR, Web Worker 등)은 **호출부를 그대로 유지**. UI 레이어만 교체. 수용: (A) 항목 코드 변경 0건.
2. **UI 마크업 교체** — 8장 (B-1)(B-2) 매핑대로 UI5 컨트롤 → HTML 요소. 수용: 화면에서 UI5 컨트롤(`sap.m.*` 등) 미생성, `.codex_tmp` 시안과 구조 일치.
3. **상태/렌더 분리** — JSONModel 바인딩 → JS 상태객체(`loginState`, `serverList`) + `render*()` 함수(8장 B). 수용: 모델 변경 시 DOM 동기화 동작.
4. **테마 토큰 소비** — 모든 색·radius·shadow 를 `theme/tokens.css` 의 의미 토큰(12번 6.2)으로. 수용: `login.css`/`serverlist.css` 내 hex 색상·하드코딩 radius·shadow **0건**(grep 검증) **+ 12번 6.6 체크리스트 표면**(스크롤바·드롭다운 펼침·focus 링·hover/disabled·ValueState·placeholder·서버 테이블 row hover/선택 등) 전수 토큰화.
5. **테마 전환 연동** — `applyTheme` 대신 `U4ATheme.apply(name)`(12번 5.2). 메타로 전달되는 `THEMEINFO.THEME`(2.4 `oMetadata`)을 `data-theme` 로 매핑(`sap_horizon→white`, `sap_horizon_dark→dark` 등 5종 키 매핑). 수용: 테마 전환 시 로그인/서버리스트 색상 함께 변경.
6. **텍스트 비하드코딩** — 메시지 키(M0271 등)는 JS 메시지 객체/JSON fetch 로(8장 B). 수용: 화면 문구 하드코딩 없음.

### 9.4 불변 제약 (반드시 보존)

- **Electron/Node 자원 전부 유지** — 8장 (A). 특히 `fnLoginPage()` 의 `REMOTE.BrowserWindow`, `oMetadata` IPC 전송, Named Pipe 라우팅, 세션 유지 워커.
- **인증·세션 로직 불변** — `ev_login()` 의 XHR `/wsloginchk`, `withCredentials`, `session.partition(SESSKEY)`, `fnServerSession` 워커. UI만 교체하고 흐름은 건드리지 않는다.
- **테마는 12번 문서 단일 출처** — 색/모양은 토큰만, 하드코딩·드리프트 금지. Preview iframe(WS20)은 UI5 테마 유지(이 화면들엔 해당 없음).
- **메타 계약 유지** — `oMetadata`(`SERVERINFO`/`THEMEINFO`/`EXEPAGE`/`SESSIONKEY`/`BROWSERKEY`) 구조와 `if-meta-info` IPC 채널명을 바꾸지 않는다.
- **기능 동등성** — 기존 기능·이벤트·단축키·검증·상태·서버분기·엣지케이스를 하나도 빠짐없이 동등하게 재현한다(외형이 아니라 행동 기준). 임의 단순화·동작 생략 금지.
